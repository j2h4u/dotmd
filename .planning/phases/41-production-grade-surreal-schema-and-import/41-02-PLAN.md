Replace the Phase 38 import proof runner with production migration execution semantics.

Purpose: The current import path proves transformability but still defaults to whole-target clearing and spike-level rollback behavior. Phase 41 needs plan, dry-run, apply, verify, and report-ready manifests that preserve stored data without default recomputation and that make overwrite/partial-failure behavior explicit.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Copied SQLite snapshot -> migration manifest | Snapshot data is trusted only as read-only migration evidence and must not trigger recomputation. |
| Graph exporter / feedback provider -> Surreal target | Exported rows can contain arbitrary identifiers and metadata that must remain data-only. |
| Operator target URL / overwrite policy -> target writes | The runner can mutate a Surreal target only after explicit mode, safety gate, and overwrite checks. |
| Migration phase checkpoints -> success report | Checkpoints decide whether a migration is safe to treat as applied or restore-required. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-06 | Tampering / Data Loss | `run_surreal_migration()` target writes | mitigate | Default overwrite policy refuses non-empty targets; explicit replacement records target pre-counts and policy before any deletion. |
| T-41-07 | Tampering | Source refs and graph identifiers | mitigate | Continue using `SurrealRecordIdCodec` for record IDs and keep raw identifiers in explicit payload fields only. |
| T-41-08 | Information Disclosure | Migration reports carrying source refs, feedback messages, and graph properties | mitigate | Reports must be written only to caller-provided paths in Plan 41-03 and should keep row samples bounded; this plan structures report data so callers can redact if needed. |
| T-41-09 | Denial of Service | Long imports over production-derived data | mitigate | Use named phases and checkpoint counts so long-running imports can fail with phase evidence; no per-request or service-start import path is added. |
| T-41-10 | Repudiation / Unsafe Rollback | Partial apply failure | mitigate | Failed phase, errors, restore_required, committed false, and phase checkpoints are mandatory on failure; no false success after rollback/cleanup. |
| T-41-SC | Tampering | Package or CLI installs | accept | No package-manager or `surreal` CLI install task is planned. If implementation discovers an install requirement, stop for human verification because research flagged `surrealdb` as suspicious. |
</threat_model>
