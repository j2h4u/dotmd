# Phase 46 Plan 01: SurrealDB Write Path and Trickle Cutover

## Objective

Close the remaining production cutover gap after Phase 45: SurrealDB retrieval
works, but writes still land only in SQLite/sqlite-vec/FalkorDB.

This plan deliberately does not start by rewriting `IndexingPipeline` to write
directly to SurrealDB. The first vertical slice is a controlled incremental sync
path from the authoritative old stack into standalone SurrealDB. That gives us
idempotency, progress, retry, and daily-update evidence before deciding whether
to inline Surreal writes into trickle.

## Decisions

- `search_backend=surreal` remains retrieval-only.
- Do not overload `search_backend` for ingestion. Any future direct Surreal
  write path needs its own config seam.
- Old-stack indexing remains the authoritative incremental detector for this
  phase: fingerprints, chunk IDs, text hashes, and vector components decide what
  changed.
- The full migration runner remains a bootstrap/recovery tool, not the daily
  update path.
- No daily path may call blanket target wipes such as schema-owned table clear,
  graph `delete_all()`, or full snapshot reload.
- A delta manifest must be derived from changed old-stack rows only. Whole-table,
  whole-source, or whole-database manifests are rejected for the daily path even
  if they are written through idempotent upserts.

## Task 1: Delta Manifest Contract

Define a small, explicit delta manifest for changed old-stack rows.

The manifest must cover at least:

- changed/deleted filesystem documents and source units;
- affected chunks, chunk-file bindings, provenance rows, resource bindings, and
  fingerprint rows;
- embeddings and vector components for changed chunk IDs only;
- graph and feedback sections must be either populated with changed rows only or
  explicitly marked `deferred`; deferred sections are not applied by this phase.

Acceptance:

- manifest shape is documented and tested;
- unchanged files do not appear in the delta, and whole-table/whole-source
  manifests are rejected;
- deleted/tombstoned refs are represented explicitly;
- manifest includes a stable checkpoint/watermark candidate but does not advance
  it until apply verification passes.

## Task 2: Idempotent Incremental Sync Runner

Implement a runner that applies a delta manifest to standalone SurrealDB.

Use the existing Surreal row shapes and stable IDs from
`ingestion/migrate_surreal.py` and `storage/surreal.py`; do not invent a second
schema translator.

Acceptance:

- apply order is deterministic: deletes/tombstones, documents/source units,
  chunks/provenance/bindings/fingerprints, embeddings/vector components, then
  any non-deferred graph/feedback sections, then checkpoint advancement;
- rerunning the same manifest is a no-op by observable counts and row content;
- a forced mid-run failure writes progress and resumes without duplicating rows;
- progress JSON includes percent, elapsed time, target size if available, and
  ETA for operations expected to exceed 120 seconds.
- instrumentation is testable with a simulated long-running phase so ETA
  formatting and heartbeat updates are exercised without needing a real
  multi-minute fixture.

## Task 3: Daily Update Smoke

Prove the sync path on a tiny controlled fixture before touching production
data.

Acceptance:

- one changed markdown file updates Surreal rows without full reindexing;
- unchanged embeddings keep the same IDs/fingerprints;
- a deleted file removes or tombstones only its affected refs;
- Surreal-backed CLI/API search can see the changed result after sync;
- the second sync run reports no meaningful work.

## Task 4: Trickle Boundary Decision

Use the evidence from Tasks 1-3 to decide the next boundary:

1. keep old-stack writes plus incremental Surreal sync for a bounded transition;
2. add a direct Surreal ingest sink after local SQLite commit;
3. block cutover until full direct writes are implemented.

Acceptance:

- decision is recorded in a Phase 46 artifact;
- production cutover criteria are updated;
- any accepted hybrid transition has an explicit drift check and rollback story.

## Verification

- Focused tests for manifest generation, idempotent apply, resume, and deletion
  semantics.
- Focused progress tests with injected clock/heartbeat or chunked fake work that
  crosses the 120-second instrumentation threshold.
- `ruff check` on changed files.
- Focused live smoke against a disposable SurrealDB database, not the production
  target.
- No production cutover until the Phase 46 decision artifact is written.

## Non-Goals

- Removing SQLite/sqlite-vec/FalkorDB.
- Rebuilding the whole production SurrealDB target from scratch.
- Changing the public MCP/API search contract.
- Solving reranker latency.
