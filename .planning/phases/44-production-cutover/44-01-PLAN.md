# Phase 44 Plan 01: Standalone SurrealDB Quality Gate and Cutover Decision

## Objective

Decide whether dotMD can move production traffic to standalone SurrealDB as the
single storage/retrieval backend.

Phase 43 proved the migration and shadow-run path. Phase 44 must convert that
evidence into a go/no-go decision by rerunning the final acceptance surface,
checking the real user-facing runtime surfaces, and defining rollback before any
live switch.

Embedded SurrealKV is not the production path for this plan. It remains rejected
until the HNSW segment-size blocker is removed upstream or by a separate,
explicit spike.

## Decisions

- Final acceptance is evaluated twice: with reranker enabled and with reranker
  disabled.
- The gate is production-surface-first, not benchmark-only: MCP, API, CLI, and
  trickle must all work against the SurrealDB-backed runtime.
- Rollback is restore-based. If cutover is approved, the old stack is not kept
  as a live fallback backend; rollback restores the pre-cutover deployment and
  data state.
- No-go is the default for unresolved quality, latency, disk, RSS, smoke, or
  restoreability risk unless explicitly accepted in writing.

## Tasks

### Task 1: Final Acceptance Comparison

Run the old stack and standalone SurrealDB candidate on the same production-
derived acceptance set.

Capture:

- quality classifications and accepted semantic differences;
- reranker-on latency;
- reranker-off latency;
- disk usage for old stack and standalone SurrealDB;
- RSS/process memory during the run.

Artifacts:

- `.planning/phases/44-production-cutover/artifacts/acceptance-reranker-on.md`
- `.planning/phases/44-production-cutover/artifacts/acceptance-reranker-off.md`
- `.planning/phases/44-production-cutover/artifacts/storage-footprint.md`
- `.planning/phases/44-production-cutover/artifacts/cutover-decision.md`

Acceptance:

- every material search difference is classified;
- every quality regression is fixed or recorded as no-go;
- latency, disk, and RSS are compared against the old stack.

### Task 2: Runtime Smoke Matrix

Smoke the real SurrealDB-backed runtime through all production surfaces:

- MCP `search` and `read`;
- API health and search/read path where available;
- CLI search/read path;
- trickle startup and one incremental update.

Artifacts:

- `.planning/phases/44-production-cutover/artifacts/smoke-mcp.md`
- `.planning/phases/44-production-cutover/artifacts/smoke-api.md`
- `.planning/phases/44-production-cutover/artifacts/smoke-cli.md`
- `.planning/phases/44-production-cutover/artifacts/smoke-trickle.md`

Acceptance:

- all four surfaces pass against standalone SurrealDB;
- no smoke depends on SQLite/sqlite-vec/FalkorDB as a hidden fallback.

### Task 3: Cutover or No-Go Record

Write the decision record.

If approved:

- define the exact production checkpoint;
- define restore commands and data paths;
- define when the old stack can be removed in Phase 45.

If not approved:

- record blocking evidence;
- define the minimum retry conditions;
- keep the old runtime as production.

Artifacts:

- `.planning/phases/44-production-cutover/artifacts/cutover-approval.md`
- `.planning/phases/44-production-cutover/artifacts/rollback-boundary.md`

Acceptance:

- the phase ends with a clear cutover/no-go decision;
- rollback can be executed without interpretation;
- Phase 45 only starts after an accepted cutover.

## Success Criteria

- Final old-vs-new acceptance is documented with reranker on and off.
- MCP, API, CLI, and trickle smoke pass on the standalone SurrealDB runtime.
- Disk and RSS are within the accepted envelope or explicitly block cutover.
- No unresolved quality regression remains unless deliberately accepted.
- Rollback is restore-based and documented.
- Embedded SurrealKV retry is not part of the main cutover path.
