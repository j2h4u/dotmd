<objective>
Add a global embedding cache (text_hash → vector BLOB) in index.db so that TEI HTTP calls
are skipped for chunks whose text has already been embedded — even after a file move that
purges the chunk from vec_meta.

Purpose: File moves currently trigger full re-embedding because vec_meta rows are deleted
during purge. The embedding_cache is keyed on text content (text_hash), not file path,
so the vector survives the purge and is reused on the next index cycle.

Output: storage/cache.py (new) with EmbeddingCache; pipeline.py modified to wire the cache
into _embed_chunks() before and after the encode_batch() call.
</objective>

  <action>
Create backend/src/dotmd/storage/cache.py as a new file.

The file contains one class: EmbeddingCache.

EmbeddingCache.__init__(self, conn: sqlite3.Connection) — stores conn, calls ensure_table().

EmbeddingCache.ensure_table() — runs:
```sql
CREATE TABLE IF NOT EXISTS embedding_cache (
    text_hash TEXT PRIMARY KEY,
    embedding BLOB NOT NULL,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
)
```
Then calls conn.commit().

EmbeddingCache.lookup(self, text_hashes: list[str]) -> dict[str, list[float]]:
- If text_hashes is empty, return {}.
- Query in batches of 500 (same limit as lookup_embeddings_by_text_hash):
  SELECT text_hash, embedding FROM embedding_cache WHERE text_hash IN (?,?,...)
- For each row, deserialize blob: dim = len(blob) // 4; vec = list(struct.unpack(f"{dim}f", blob))
- Return {text_hash: vec} dict.
- On any exception, log warning and return {} (graceful degradation).

EmbeddingCache.store(self, text_hash: str, embedding: list[float]) -> None:
- Serialize: blob = struct.pack(f"{len(embedding)}f", *embedding)
- INSERT OR IGNORE INTO embedding_cache (text_hash, embedding) VALUES (?, ?)
  Use INSERT OR IGNORE so duplicate text_hashes are silently skipped (idempotent).
- Do NOT commit here — caller commits in bulk after the encode loop.

Imports needed: from __future__ import annotations, import logging, import sqlite3,
import struct. No other imports.

Note: text_hash values in the production DB are a mix of 32-char MD5 (legacy rows) and
128-char blake2b (current pipeline). TEXT PRIMARY KEY handles both lengths correctly.
New entries written by the current pipeline will always be blake2b-128.
  </action>
