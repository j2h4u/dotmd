<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| package registry -> repo dependency | PyPI metadata and source links influence a repo dependency change. |
| copied snapshot -> Surreal embedded store | Existing production-derived data is transformed into a new database engine. |
| test fixtures -> migration evidence | Fixture evidence may be mistaken for production parity if not labeled. |
| query builder -> SurrealDB | Migration and storage code sends SurrealQL and record IDs to the database engine. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-38-02-SD | Tampering | import source snapshots | mitigate | Depend on Plan 38-01 snapshot manifest; import refuses incomplete or drifted inventory categories. |
| T-38-02-SQLI | Tampering | Surreal query construction | mitigate | Use parameter binding for values and a central record-ID sanitizer for identifiers; tests include malicious query text and record IDs. |
| T-38-02-SECRETS | Information Disclosure | package verification and import reports | mitigate | Record URLs and package metadata only; do not copy tokens, env values, or container secrets into reports. |
| T-38-02-PARTIAL | Tampering / Availability | transform import apply mode | mitigate | Import report records dry-run counts, committed counts, rollback status, and category-level failures; partial apply cannot be marked successful. |
| T-38-02-PARITY | Integrity | storage model completeness | mitigate | This plan proves representability and transform import only; Plan 38-03 decides retrieval parity. |
| T-38-02-SC | Tampering | `surrealdb` package install | mitigate | Planning-time human approval is recorded; executor verifies PyPI/source evidence before modifying `backend/pyproject.toml`. |
</threat_model>
