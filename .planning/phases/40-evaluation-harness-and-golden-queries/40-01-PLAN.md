<objective>
Create the Phase 40 quality evaluation surface for the SurrealDB-native cutover.

Purpose: Phase 39 defined that search quality is judged through user-visible accepted differences instead of exact old-stack rank parity. This plan turns that contract into a test-covered corpus, typed evaluator, and report runner so later migration phases have a concrete quality gate.

Output: A typed evaluation module, a JSONL golden corpus plus review ledger, a devtool runner that emits machine-readable JSONL diff rows and a concise Markdown summary, focused tests, and durable documentation. Phase 40 keeps the old stack as baseline/evaluator evidence only and does not implement real Surreal schema/import/retrieval, production shadow execution, reindexing, TEI reembedding, runtime fallback, or compatibility mode.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| JSONL corpus / result files -> evaluator | Local files influence cutover gate evidence and must be parsed strictly. |
| Captured old-stack / Surreal result rows -> diff classifier | Result rows are evaluator inputs, not trusted runtime state. |
| Devtool CLI arguments -> filesystem report writer | Output paths and input paths come from the operator running the harness. |
| Acceptance metadata -> aggregate gate | Acceptance can resolve an otherwise blocking diff and must be explicit. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-01 | Tampering | `load_golden_queries()` / `load_eval_results()` | mitigate | Validate required JSONL fields, duplicate ids, enum values, category coverage, and UTF-8 one-row-per-line structure before classification. |
| T-40-02 | Tampering | `summarize_diffs()` acceptance handling | mitigate | Require `accepted_by` and `accepted_reason` for accepted regression or unclear rows, preserve raw `AcceptedDifference` and `CutoverGate`, and count unresolved rows separately. |
| T-40-03 | Information Disclosure | `surreal_eval_runner.py` report writer | mitigate | Write only to explicit report paths, avoid reading arbitrary content from label refs during report generation, and include refs/snippets only from supplied result rows. |
| T-40-04 | Tampering | query text in CLI/devtool flow | mitigate | Keep query execution outside shell commands; tests must prove the runner consumes JSONL through Python code and does not shell-interpolate query text. |
| T-40-05 | Denial of Service | large or malformed JSONL inputs | mitigate | Stream JSONL row-by-row, fail fast with line-numbered `ValueError`, and keep focused tests on malformed row handling. |
| T-40-SC | Tampering | package installs | accept | No npm, pip, or cargo install task is planned; Phase 40 uses Python stdlib and existing project dev dependencies. |
</threat_model>
