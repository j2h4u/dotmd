# Phase 999.12: Dual-encoder unified embedding — decoupled metadata vectors - Context

**Gathered:** 2026-04-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Decouple embedding of chunk body text from embedding of document metadata (title, tags) so that frontmatter changes trigger O(1) TEI calls per file instead of O(N chunks). Introduce N-vector weighted fusion architecture with query logging as observability foundation.

**In scope:**
- New `vec_components` table for storing raw component vectors as BLOBs (text per chunk, meta per file)
- e_fused replaces current enriched vectors in vec0; computed from weighted sum of components
- `DOTMD_EMBEDDING_WEIGHTS` env var for weight configuration
- `search_log` table in index.db for query logging (observability + future calibration foundation)
- Schema version bump + wipe-on-deploy migration strategy
- Separate meta_tracker fingerprint (title+tags) from chunk_tracker (body+kind)

**Out of scope:**
- Auto-calibration (deferred to Phase 999.15 and 999.16)
- pgvector migration (deferred to Phase 999.14)
- Separate title vs tags as independent components (post-eval)
- Temporal embeddings
- Per-kind weights (post-eval)

</domain>

<decisions>
## Implementation Decisions

### Storage Architecture

- **D-01:** N-vector scheme from day one. All embedding components stored in a single table `vec_components_<strategy>_<model>(entity_id TEXT, component TEXT, embedding BLOB NOT NULL, PRIMARY KEY (entity_id, component))`. `text` component keyed by chunk_id; `meta` and other file-level components keyed by file_path.
- **D-02:** e_fused lives in existing vec0 virtual table — ANN search preserved. vec0 holds only e_fused (not raw components). Raw components are BLOB in regular SQLite (`vec_components`) — writable via standard SQL UPDATE.
- **D-03:** sqlite-vec stays. Rationale: DELETE+INSERT for ~8 vec0 rows per file on metadata change is not meaningful overhead; atomic transactions with FTS5/metadata in single index.db is a real advantage; all vectors rebuilt on deploy anyway. pgvector deferred to Phase 999.14.
- **D-04:** On metadata change: read `e_text` BLOB from `vec_components` for each chunk of the file → compute 1 new `e_meta` (1 TEI call) → recompute `e_fused` locally → DELETE+INSERT ~8 rows in vec0. No full file reindex.

### Fusion Formula

- **D-05:** N-vector weighted fusion: `e_fused = normalize(Σᵢ wᵢ · normalize(eᵢ))` where weights sum to 1. Generalizable to any number of components — new components are additive rows in `vec_components`, no schema migration.
- **D-06:** Initial components: `text` (per chunk) + `meta` (per file = title+tags combined in one embed). Default weights: text=0.7, meta=0.3.
- **D-07:** e_meta = embed(title + tags as single string) — not separated. One TEI call per file per metadata change. Splitting title/tags into independent components deferred until post-eval evidence shows benefit.

### Weights Configuration

- **D-08:** `DOTMD_EMBEDDING_WEIGHTS` env var, comma-separated key=value format: `text=0.7,meta=0.3`. Follows DOTMD_* convention. Must sum to 1.0 (validate at startup).
- **D-09:** Changing weights is cheap (local math, no TEI) — recomputes e_fused for all chunks, updates vec0. Can be applied without full reindex.
- **D-10:** Auto-calibration NOT in this phase. Deferred: cross-encoder signal → Phase 999.15; cross-engine agreement signal → Phase 999.16.

### Query Logging

- **D-11:** Add `search_log` table to index.db: `(id INTEGER PRIMARY KEY, query TEXT, timestamp TEXT, weights_used TEXT, top_results TEXT)`. `top_results` stores JSON array of `{chunk_id, score, engine}` for top-N results.
- **D-12:** Logging is the foundation for 999.15/999.16 auto-calibration and general observability. Log every search request. Retention policy: TBD in planning (consider rolling window or max row count).

### text_hash Migration

- **D-13:** text_hash semantics change: from `hash(title + tags + chunk_text)` → `hash(chunk_text only)`. Old enriched-text hashes are semantically incompatible — old cached vectors are `embed(title+tags+chunk_text)`, new e_text must be `embed(chunk_text only)`. Cannot reuse old vectors even by re-keying.
- **D-14:** Migration strategy: **wipe on deploy**. Schema version bump in `vec_config` triggers full invalidation of `embedding_cache` and `vec_meta_*` on container startup. trickle rebuilds from scratch. Expected rebuild time: **several days** on current hardware (Xeon E3-1245 V2, TEI CPU-only). Plan accordingly — search quality degrades for days after deploy.
- **D-15:** Separate fingerprint trackers: `chunk_tracker` = hash(body+kind) unchanged; new `meta_tracker` = hash(title+tags) per file, replaces the title+tags portion of the old `embed_tracker`. Trickle detects metadata-only changes via meta_tracker and triggers cheap e_meta recompute + e_fused update (not full re-chunk).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `FileTracker` class (pipeline.py) — already supports multiple trackers per strategy/model. meta_tracker follows exact same pattern as chunk_tracker and embed_tracker.
- `EmbeddingCache` / `embedding_cache` table — exists, just needs invalidation on schema version bump. No structural change needed.
- `vec_config_<strategy>_<model>` table — already stores `(key, value)` pairs per strategy+model. Use this to store schema version sentinel for migration detection.
- `SemanticSearchEngine.encode_batch()` — reusable for encoding both e_text (chunks) and e_meta (file metadata string). Same TEI call, different inputs.

### Established Patterns
- Two-dimensional table naming: `<table>_<strategy>_<model>` — `vec_components` must follow this: `vec_components_<strategy>_<model>`.
- Migration pattern: increment schema version in `vec_config`, run migration in `_ensure_schema()` on startup. See `migration_v16.py` for reference.
- `content_handlers.py` registry pattern — `enrich_with_title_and_tags` is called via handler registry, not directly. Removal must go through registry.

### Integration Points
- trickle's metadata-change path (pipeline.py ~1080–1132): currently calls `_embed_chunks(chunks)` on metadata change. After 999.12: reads e_text BLOBs from `vec_components`, calls TEI once for e_meta, recomputes e_fused, updates vec0.
- Search path: `SemanticSearchEngine.search()` queries vec0 — unchanged. vec0 now holds e_fused instead of enriched embedding, but the query interface is identical.
- `lookup_embeddings_by_text_hash()`: after migration, text_hash = hash(chunk_text only). Cross-strategy reuse now truly works (same chunk text in different strategies = same e_text = same hash = 1 TEI call total).

</code_context>

<specifics>
## Specific Ideas

- The user explicitly described fusion weights as "attractors" — each component vector pulls e_fused toward its semantic region of embedding space. This framing is useful for explaining the system.
- Rebuild time after deploy is **days, not hours** on current hardware. This is a known trade-off, not a bug. Plan deployment accordingly (weekend deploy, or accept degraded search for days).
- N-vector abstraction was deliberately designed in from the start so future components (separate title, separate tags, temporal signal) can be added as new rows in `vec_components` without schema migration.
- The user wants query logging primarily for future auto-calibration foundation and general observability/traceability — not for immediate use.

</specifics>
