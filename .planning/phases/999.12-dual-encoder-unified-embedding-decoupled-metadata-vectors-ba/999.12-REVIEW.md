# Phase 999.12: Code Review Report

**Reviewed:** 2026-04-27
**Depth:** standard
**Files Reviewed:** 14
**Status:** issues_found

## Summary

Phase 999.12 introduces a dual-encoder unified embedding architecture: chunk body text (`e_text`) and file metadata (`e_meta`) are now encoded as separate component vectors stored in `VecComponentStore`, then fused into `e_fused` before being written to the sqlite-vec `vec0` table. A two-fingerprint change detection scheme (`chunk_tracker` / `meta_tracker`) separates full re-chunk paths from cheap metadata-only paths (1 TEI call).

The architecture is sound and the core design — canonical path normalization via `_meta_entity_id()`, two-phase compute-then-write, dimension-mismatch raise — is well-executed. Most prior review concerns have been addressed.

However, one correctness bug was found: the schema-version wipe path is explicitly documented as non-atomic ("each sub-operation commits internally") but then tries to guarantee crash safety with "sentinel is written last." This guarantee is only partial — a crash between any two of steps 1–6 leaves the database in an inconsistent state that re-wipe on next startup does not fully recover because sub-operation deletes are already committed. This is a BLOCKER.

Five additional warnings cover real-but-non-crashing defects: a `_check_schema_version` path that silently skips migration on a pre-999.12 database, a race window in the metadata-only fingerprint path, the `_build_file_meta_from_fileinfo` null-guard gap, an `add_chunks(overwrite=True)` in `_check_weights_changed` that wipes ALL vectors on every call, and a `_save_meta_fingerprint` call that escapes the no-chunks early-return guard in `_index_file_embed`.

## Critical Issues

### CR-01: Schema-version wipe claims crash safety it cannot deliver — partial wipes corrupt index state permanently

**File:** `backend/src/dotmd/ingestion/pipeline.py:2217-2244`

**Issue:** `_check_schema_version()` wipes 7 state components across multiple stores. The docstring correctly states that each sub-operation "commits internally" and therefore an outer `BEGIN/COMMIT` cannot wrap them. It then claims crash safety via "sentinel is written last." That claim is false: if the process crashes after step 2 (`vec_components.delete_all()`) but before step 3 (`vector_store.delete_all()`), `vec0` still contains stale pre-999.12 enriched vectors while `vec_components` is empty. On next startup, `stored_version` will still be the old value, the wipe runs again — but `vec_components.delete_all()` is a no-op (already empty), and `vector_store.delete_all()` succeeds. The `chunk_tracker.clear()` and `meta_tracker.clear()` in steps 4–5 then force a full re-embed. This means re-embed on next trickle run starts writing new `e_fused` vectors into `vec0` — but `vec_components` is still empty because no completed indexing run has stored `e_text` BLOBs yet. So the next metadata-only fast path will find zero `e_text` BLOBs and fall back to full re-embed correctly. So the _eventual_ outcome is safe, but the intermediate window (vec0 stale, components empty, chunk/meta trackers not yet cleared) can cause trickle to search against stale enriched vectors until the full re-embed completes — which may take days. More critically: if the crash happens between step 5 (meta_tracker.clear) and step 6 (DELETE weights_used), trickle's weight check on next run fires against stale vec0 content and silently stores a wrong weights_used sentinel, blocking future weight-change detection.

The deeper issue: `_check_schema_version` is documented to guarantee "if COMMIT is not reached, next startup re-wipes." But at `stored_version != SCHEMA_VERSION` the re-wipe path runs every startup until step 7 is committed. Steps 1–6 are idempotent in isolation (double-deletes are no-ops) but `self._conn.commit()` at line 2242 covers only the `DELETE weights_used` + `INSERT schema_version` pair — not the preceding individual commits. So the sentinel guarantee holds for steps 6–7, not for steps 1–5.

**Fix:** Two options:

Option A (minimal): Accept non-atomicity but remove the false crash-safety claim. Document explicitly that a crash mid-wipe is tolerated because each step is idempotent and the sentinel write is last. Add a guard: if the sentinel write fails (exception), log CRITICAL and do not suppress — the operator must restart manually.

Option B (preferred): Move the sentinel check to the BEGINNING of the wipe and write a "wipe_in_progress" sentinel at the start. On next startup, if "wipe_in_progress" is present, repeat all wipe steps before writing SCHEMA_VERSION. This gives true crash-safe wipe semantics:

```python
# Step 0: write in-progress sentinel first (marks uncommitted state)
self._conn.execute(
    f"INSERT OR REPLACE INTO {config_table} (key, value) VALUES ('schema_version', 'WIPE_IN_PROGRESS')"
)
self._conn.commit()

# Steps 1-6: idempotent wipes ...

# Step 7: write final sentinel (marks clean state)
self._conn.execute(
    f"INSERT OR REPLACE INTO {config_table} (key, value) VALUES ('schema_version', ?)",
    (self.SCHEMA_VERSION,),
)
self._conn.commit()

# In _check_schema_version(), treat 'WIPE_IN_PROGRESS' == wipe path, not None path.
```

## Warnings

### WR-01: `_check_schema_version` silently skips migration for pre-999.12 databases — existing corrupted indexes not wiped

**File:** `backend/src/dotmd/ingestion/pipeline.py:2199-2209`

**Issue:** When `stored_version is None` (line 2199), the code writes the new sentinel and returns without wiping. The comment says this handles "fresh DB or pre-999.12 DB that predates SCHEMA_VERSION tracking." For a fresh DB this is correct — there is nothing to wipe. But for a pre-999.12 database that already has data (stale enriched `e_fused` in `vec0`, stale `embedding_cache` with body+title+tags hashes), skipping the wipe causes the pipeline to treat existing stale vectors as valid. The fast-path (`_embed_existing_chunks`) will find no `e_text` BLOBs in `VecComponentStore` (they did not exist before 999.12), log a warning, and fall back to full re-embed for each file — which is the correct behavior. However, stale `embedding_cache` entries keyed on pre-999.12 `text_hash = hash(body+prefix)` will not be reused (key format changed), so they are dead weight but not actively harmful.

The real risk: if the pre-999.12 database has a `vec_components` table from an aborted partial run of 999.12, `VecComponentStore` may have stale partial entries. Skipping the wipe lets those stale entries be used by `_embed_chunks()` Layer 1 cache, returning wrong vectors without any warning.

**Fix:** Add a check: if `stored_version is None` AND `vec_components.count() > 0`, treat it as the wipe path (not the fresh-DB path), since no valid pre-999.12 deployment could have `vec_components` rows:

```python
if stored_version is None:
    if self._vec_components.count() == 0:
        # Fresh DB — write sentinel and continue.
        self._conn.execute(...)
        self._conn.commit()
        return
    # Pre-999.12 DB with partial vec_components data — fall through to wipe.
    stored_version = "PRE_999.12"  # trigger wipe path below
```

### WR-02: `_index_file_embed` saves meta fingerprint for the no-chunks early-return case — file incorrectly marked as embedded

**File:** `backend/src/dotmd/ingestion/pipeline.py:788-790`

**Issue:** When `chunks` is empty after loading from the metadata store, the method calls `self._save_meta_fingerprint(file_info)` (line 789) and returns. This marks the file as having an up-to-date meta fingerprint even though no embedding was computed or stored. On the next run, `meta_tracker.diff()` will see the saved fingerprint and classify the file as `unchanged` — so the pipeline will not re-attempt embedding even if the file later gets chunks (e.g. after a trickle re-chunk). This is incorrect: a file with no chunks should not be marked as having a current meta embedding.

```python
if not chunks:
    logger.debug("_index_file_embed: no chunks for %s", file_info.path)
    self._save_meta_fingerprint(file_info)  # ← BUG: marks as embedded when nothing was done
    return
```

**Fix:** Remove the `_save_meta_fingerprint` call from the no-chunks early-return path. Let the meta fingerprint remain absent/stale; the next `meta_tracker.diff()` will re-classify the file as needing embedding once it has chunks:

```python
if not chunks:
    logger.debug("_index_file_embed: no chunks for %s — skipping (will retry after chunks arrive)", file_info.path)
    return
```

### WR-03: `_save_and_embed_chunks` does not commit `VecComponentStore` stores before calling `add_chunks` — data loss window in autocommit mode

**File:** `backend/src/dotmd/ingestion/pipeline.py:1243-1265`

**Issue:** In `_save_and_embed_chunks`, the per-chunk `self._vec_components.store(chunk.chunk_id, "text", e_text)` calls at lines 1243–1244 do not call `commit()`. The connection is in `isolation_level=None` (autocommit), so individual `INSERT OR REPLACE` statements auto-commit as soon as they are issued. This is fine in the normal case. However, the docstring for `VecComponentStore.store()` explicitly says "does NOT call commit() — the caller owns the transaction boundary." In WAL autocommit mode, individual statement commits happen implicitly after each statement, so there is no explicit transaction wrapping these stores. If `_vector_store.add_chunks()` at line 1262 then issues `DELETE FROM vec_chunks` (the `overwrite=True` path at `sqlite_vec.py:227`) and then fails mid-loop inserting rows into `vec0`, the `VecComponentStore` rows for those chunks are already committed but the corresponding `vec0` rows are partially written. The next startup has orphaned `e_text` BLOBs in `vec_components` with no corresponding `vec0` rows.

This is not a crash — `add_chunks` commits at the end of its loop (line 250) and individual row inserts auto-commit — but the delete+re-insert in `add_chunks` is non-atomic relative to `VecComponentStore` writes. An OOM or signal during `add_chunks`'s insert loop can leave `vec_components` ahead of `vec0`.

**Fix:** Flush `VecComponentStore` stores with an explicit commit before calling `add_chunks`, matching the pattern already used in `_embed_existing_chunks` (line 2105):

```python
# Store e_text in VecComponentStore for future fast-path reads
for chunk, e_text in zip(file_chunks, e_text_vectors):
    self._vec_components.store(chunk.chunk_id, "text", e_text)
# Store e_meta component
e_meta = self._embed_meta_component(fi)
# Flush VecComponentStore stores before vec0 write (matches _embed_existing_chunks pattern)
self._conn.commit()
# Fuse per chunk
e_fused = [...]
```

### WR-04: `_check_weights_changed` calls `add_chunks(overwrite=True)` — wipes ALL vectors in the table on first file, then appends for the rest

**File:** `backend/src/dotmd/ingestion/pipeline.py:2341`

**Issue:** Inside the per-file loop, `self._vector_store.add_chunks(chunks, e_fused, overwrite=True)` is called for every file. In `SQLiteVecVectorStore.add_chunks`, `overwrite=True` triggers `DELETE FROM vec_chunks` and `DELETE FROM vec_meta` (lines 227–228 of `sqlite_vec.py`) before inserting. This means on the first file in the batch, all existing vectors are wiped. On the second file, `INSERT OR IGNORE` is used (line 235 of `sqlite_vec.py`) so the first file's rows are not deleted again — but there is still a window where `vec0` contains only the first file's fused vectors if a crash occurs between files. More importantly: for a large index with 1000s of files, the `DELETE FROM vec_chunks` on every file is extremely slow (each call is a full table scan on the virtual table), and it means no file processed before the current file remains in `vec0` during the iteration.

In practice with WAL and no crash this works correctly but is very inefficient and semantically wrong (`overwrite=True` per-file is not the intended semantics here — the intent is to recompute all fused vectors, not to overwrite the whole table per file).

**Fix:** Use `overwrite=False` in the per-file loop, and issue a single `delete_all()` + reinitialize before the loop starts:

```python
# Wipe vec0 once before the loop
self._vector_store.delete_all()

# Then use overwrite=False per file
self._vector_store.add_chunks(chunks, e_fused, overwrite=False)
```

### WR-05: `_build_file_meta_from_fileinfo` accesses `fi.frontmatter` without None guard — crashes on FileInfo with `frontmatter=None`

**File:** `backend/src/dotmd/ingestion/pipeline.py:1281`

**Issue:** `_build_file_meta_from_fileinfo` calls `fi.frontmatter.get("tags", [])` at line 1281 without checking whether `frontmatter` is `None`. `FileInfo` objects constructed by `reindex_vectors()` (lines 912–915) may have `frontmatter={}` (not None) in the happy path, but the except clause at line 914 sets `fm = {}` and passes it — so that specific path is safe. However, `index_file()` constructs a `FileInfo` with `_fm = {}` when the file cannot be read (line 1402). Other callers in tests use `FileInfo(path=..., frontmatter=fm)` with an explicit `fm` dict. The crash risk exists if any code path constructs a `FileInfo` without a `frontmatter` argument and `FileInfo.frontmatter` defaults to `None` rather than `{}`.

Checking the `FileInfo` model would confirm the default; given that `_embed_meta_component` already guards with `if file_info.frontmatter` (line 713) and `reindex_vectors` guards with `if fi.frontmatter` (line 921), the inconsistency is clear — some call sites guard, `_build_file_meta_from_fileinfo` does not.

**Fix:** Add the same guard pattern used elsewhere:

```python
def _build_file_meta_from_fileinfo(self, files: list[FileInfo]) -> dict[str, tuple[str, str]]:
    file_meta: dict[str, tuple[str, str]] = {}
    for fi in files:
        fm = fi.frontmatter or {}
        tags = fm.get("tags", [])
        tags_str = ", ".join(str(t) for t in tags) if tags else ""
        file_meta[str(fi.path)] = (fi.title, tags_str)
    return file_meta
```
