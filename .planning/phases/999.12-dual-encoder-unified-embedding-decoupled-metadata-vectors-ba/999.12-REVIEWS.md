## Codex Review

### Summary

The three plans are coherent and materially aligned with the phase goal: metadata-only updates should stop invalidating every chunk embedding and should require only one metadata embedding plus local vector fusion. The current revision addresses the prior high-risk gaps much better than the earlier cycle: the branching logic, transaction ownership, `_embed_chunks()` purity, missing component fallback, bulk metadata batching, and weight-change recompute path are all now explicitly planned. ### Strengths

- The core architecture directly addresses the incident root cause by separating chunk text vectors from metadata vectors.
- The N-vector component table is simple, SQLite-friendly, and avoids introducing a new vector backend.
- Keeping `vec0` as the fused search vector preserves the current search surface while changing the indexing internals.
- The wipe-on-deploy decision is pragmatic given the changed `text_hash` semantics.
- Explicit transaction ownership (`_index_file_embed` with BEGIN/COMMIT/ROLLBACK) is a good fit because component writes, vec0 writes, fingerprint updates, and fallback behavior need to stay consistent.
- The planned tests focus on the actual invariant: metadata-only reindexing should avoid chunk re-embedding.
- The weight-change path is correctly treated as local recomputation rather than a re-embedding event.
- `_embed_meta_component()` stores to VecComponentStore but does NOT commit — respects caller-owns-transaction contract consistently.

### Concerns

- **HIGH: `entity_id` stability for metadata vectors may be fragile.**
  The metadata `meta` component is keyed by `str(file_path)`. If the same file can appear as an absolute path in one path and a relative path in another (e.g., bulk `run()` vs trickle `index_file()`), or if path normalization differs between entry points, the fast path can miss existing `e_meta` or leave stale rows. Path renames also strand old `e_meta` rows unless orphan cleanup is tied to document identity. The plan should specify: is `str(file_path)` always the resolved absolute path? Is it the same string used by `meta_tracker` and `vec_meta`?

- **HIGH: The metadata-only fallback can silently invalidate the "exactly 1 TEI call" test invariant.**
  The plan says metadata-only with missing `e_text` falls back to full embedding for missing chunks. This is correct for correctness, but the canonical invariant test (`test_metadata_only_reindex_exactly_one_tei_call`) asserts exactly 1 call after initial indexing. If run on a freshly wiped system (e.g., first run post schema-version bump), there are no `e_text` BLOBs and the "fast path" triggers the fallback. The test should document this precondition explicitly, or add a separate test for the "healthy steady-state fast path" vs "first-run fallback path."

- **MEDIUM: `chunk_tracker` and `meta_tracker` fingerprint save ordering.**
  `_index_file_embed()` saves the meta fingerprint in both body and metadata cases. But the plan does not specify when the `chunk_tracker` fingerprint is saved. If `chunk_tracker` is saved before `_index_file_embed()` is called and the transaction rolls back, the file will appear body-indexed but have no fused vectors, causing a silent miss on next run. The plan should confirm that chunk fingerprint saves are inside the same transaction boundary or happen only after `_index_file_embed()` succeeds.

- **MEDIUM: Schema wipe atomicity.**
  The plan says write `schema_version='2'` last (good), but the wipe steps and sentinel write are not explicitly wrapped in a single transaction. If interrupted after clearing `embedding_cache` but before writing the sentinel, the next startup re-wipes (correct). But if interrupted after writing the sentinel but before clearing a component (possible if order is wrong), the system considers itself up-to-date with stale state. Confirm the entire `_check_schema_version()` body runs in one `BEGIN ... COMMIT`.

- **MEDIUM: `_check_weights_changed()` startup recompute behavior when components are missing.**
  If some `text` or `meta` components are missing (partial schema wipe, interrupted build), the plan needs clear behavior. The plan currently implies "log warning + continue," but does not specify whether the `weights_used` sentinel is updated when the recompute was only partial. A partial weights-sentinel update could make the system believe weights are current when many fused vectors are still stale.

- **MEDIUM: Vector dimension mismatch is documented as acceptable zip truncation.**
  Plan 03's dimension mismatch test says "documents zip truncation as known behavior" and asserts the result is truncated to `min(3, 2) = 2`. For embeddings, silent truncation is almost certainly data corruption — `e_text` and `e_meta` must always come from the same TEI model with the same output dimension. `_fuse_vectors()` should raise a `ValueError` on dimension mismatch, not truncate silently.

- **MEDIUM: Weight config validation does not require specific component names.**
  The `validate_embedding_weights` validator checks sum ≈ 1.0 but accepts any component names. Accepting `foo=0.3,text=0.7` would pass validation but silently omit `meta` from fusion. The validator should require both `text` and `meta` keys.

- **LOW: `search_log(id)` index is redundant.**
  `id INTEGER PRIMARY KEY` already creates efficient rowid access in SQLite. The explicit index is harmless but unnecessary.

- **LOW: `log_search()` non-fatal try/except may mask integration errors during tests.**
  Tests should confirm search_log write behavior. Plan 03 lists 5 test files but `test_search_log.py` from the RESEARCH.md is not in the list — confirm whether it was dropped or merged.

- **LOW: `embed_checksum` removal should be preceded by a repo-wide import check.**
  Plan 02 only checks `pipeline.py` imports. An `rg "embed_checksum"` across the full repo before deletion would catch any other importers.

### Suggestions

- Normalize `entity_id` for metadata components — document whether `str(file_path)` is always absolute/POSIX-normalized/resolved. Use the same canonical form as `meta_tracker` and `vec_meta`.
- Make `_fuse_vectors()` validate equal dimensions and raise a `ValueError`. Update the test from "zip truncation is known behavior" to "dimension mismatch raises."
- Require both `text` and `meta` keys in weight validation.
- Wrap `_check_schema_version()` wipe + sentinel write in a single `BEGIN ... COMMIT`.
- `_check_weights_changed()`: do NOT update the `weights_used` sentinel if the recompute was only partial (some files skipped due to missing components).
- Add precondition documentation to the 1-TEI-call invariant test.
- Run `rg "embed_checksum"` across the full repo in Plan 02 before deletion.
- Confirm `test_search_log.py` status (include or explicitly drop).

### Risk Assessment

**Overall risk: MEDIUM.** The plan is directionally sound and now covers the main prior-cycle failure modes. The remaining risks are implementation precision issues. The highest single risk is `entity_id` path normalization inconsistency, which could silently prevent the fast path from working in production while passing all tests (which use `tmp_path` consistently).

### Plan 01 — Foundation: VecComponentStore, meta_checksum, weights config

#### Summary

Solid foundation plan that introduces three independent, non-interacting primitives. The VecComponentStore follows existing sqlite-vec table naming conventions, the meta_checksum correctly isolates title+tags from body+kind, and the weights config uses Pydantic validation at startup. Low risk — no existing code paths are modified, only additions.

#### Strengths

- Table naming `vec_components_<strategy>_<model>` matches the two-dimensional convention used by `vec_chunks`/`vec_meta`/`vec_config`
- `store()` does NOT commit — matches the caller-owns-transaction pattern used throughout the codebase
- `get_batch()` batches at 500 — same limit used by `lookup_embeddings_by_text_hash` at `sqlite_vec.py:393`
- Composite PK `(entity_id, component)` with INSERT OR REPLACE gives clean upsert semantics
- Key convention (text→chunk_id, meta→file_path) correctly reflects the N:1 relationship between chunks and documents
- `meta_checksum` excluding `kind` is correct — kind changes trigger re-chunking via `chunk_checksum`
- Tags sorted before hashing prevents order-sensitivity — matches existing pattern in `embed_checksum`
- Weight validation at startup via Pydantic `field_validator` with ±0.001 tolerance is a fail-fast safety net

#### Concerns

- **[LOW]** Empty frontmatter in `meta_checksum`: must handle `title=None` and `tags=None` gracefully. The existing `embed_checksum` uses `str(frontmatter.get("title", ""))` — `meta_checksum` must match this defensive pattern.

#### Risk Assessment: **LOW**

Three independent additions with no existing code modifications.

### Plan 02 — Pipeline Refactor

#### Summary

The critical plan. The design is architecturally sound — the three-case branching in `_index_file_embed`, the schema version wipe, and the explicit transaction boundaries are all well-conceived. However, there is one significant correctness issue in the three-layer cache ordering inside `_embed_chunks()`, and several paths that need explicit specification.

#### Strengths

- Case branching is clean and complete: the three cases in `_index_file_embed` map directly to the three tracker states (body_changed / metadata_only / neither). Missing e_text BLOB fallback in Case 2 is a good safety net for partial-write recovery
- Transaction boundaries match existing code: BEGIN/COMMIT/ROLLBACK matches `index_file()` at `pipeline.py:971-978` and `_purge_file()` at `pipeline.py:1267-1275`
- Noop enrichment is correct: FTS5 uses `_build_file_meta_from_fileinfo`, not `handler.enrich`. Replacing registry entries with noop only affects the embedding path
- Schema wipe clears all 7 components: comprehensive. Sentinel written LAST, so a crash mid-wipe triggers re-wipe on next startup
- Weight-change recompute is pure math: no TEI calls, just read stored components → fuse → write vec0. Idempotent
- Bulk e_meta batching: collapsing N files × 1 e_meta into a single `encode_batch()` call is the right optimization

#### Concerns

- **[HIGH] CONCERN-01: Three-layer cache ordering returns e_fused instead of e_text**

  Plan specifies: "vec_meta text_hash → global embedding_cache → VecComponentStore get_batch"

  `lookup_embeddings_by_text_hash` (`sqlite_vec.py:362-429`) JOINs vec_meta with vec0 and returns the **vec0 embedding**. After the dual-encoder change, vec0 stores **e_fused** (not e_text). The current code uses `setdefault()`, so vec_meta hits **win** over embedding_cache hits.

  **Concrete failure scenario:**
  1. File A embedded: chunk text "budget discussion" → e_text_A → fused with e_meta_A → e_fused_A stored in vec0
  2. File B modified, has same chunk text "budget discussion" → text_hash matches
  3. `_embed_chunks()` looks up vec_meta → finds text_hash → returns **e_fused_A** (not e_text_A)
  4. Caller fuses e_fused_A with e_meta_B → double-fused vector
  5. Search quality degrades for this chunk silently

  This affects every chunk whose body text appears in more than one file — exactly what the M2M content-addressed schema supports.

  **Fix:** Remove vec_meta text_hash lookup from `_embed_chunks()`. Use VecComponentStore as primary source (stores raw e_text), embedding_cache as secondary, TEI as fallback:
  ```
  Layer 1: VecComponentStore.get_batch(chunk_ids, component='text') → e_text BLOBs
  Layer 2: embedding_cache.lookup(text_hashes) → e_text by content hash
  Layer 3: TEI encode_batch → compute new e_text
  ```

- **[MEDIUM] CONCERN-02: `_embed_existing_chunks()` not explicitly addressed**

  The method at `pipeline.py:1576-1607` handles embed-only files (model switch, etc.). It currently calls `_embed_chunks()` and `add_chunks()`. After the dual-encoder change, it must also compute e_meta per file, fuse e_text + e_meta, and write all three components. Plan Task 4 says "Update bulk run()" but doesn't name this method.

  **Fix:** Add explicit mention of `_embed_existing_chunks()` in Task 4.

- **[MEDIUM] CONCERN-03: `_save_and_embed_chunks()` must fuse before vec0 write**

  The bulk path at `pipeline.py:772-802` currently calls `_embed_chunks()` → `add_chunks()`. After the change, `add_chunks()` must receive e_fused (not e_text). The plan mentions "Update bulk run()" but doesn't specify the fusion step in this code path.

  **Fix:** Task 4 should explicitly state that `_save_and_embed_chunks()` must: (1) call `_embed_chunks()` for e_text, (2) batch e_meta across files, (3) fuse per chunk, (4) write e_fused to vec0, (5) store e_text + e_meta in VecComponentStore.

- **[MEDIUM] CONCERN-04: `_check_weights_changed()` must handle empty store and batch the recompute**

  On fresh install, VecComponentStore is empty — the recompute should skip entirely. Additionally, reading ALL e_text + e_meta from VecComponentStore for ~1.4M chunks (1024-dim each ≈ 5.6GB) could OOM on 16GB RAM. The recompute must be batched (e.g., 1000 chunks at a time: read e_text + e_meta → fuse → write e_fused → insert new vec0 rows).

  **Fix:** Add guard: `if vec_components.count() == 0: return`. Process in batches of ~1000 chunks.

- **[LOW] CONCERN-05: `embed_tracker` public property rename**

  The property at `pipeline.py:1643-1646` is used by `service.py`. Renaming to `meta_tracker` is correct but must update all callers. The plan doesn't mention this property explicitly.

#### Suggestions

- After fixing CONCERN-01, `lookup_embeddings_by_text_hash` becomes unused in the embed path — mark as deprecated or note if used elsewhere
- Add a comment in `_embed_chunks()` explaining WHY vec_meta text_hash lookup was removed (future developers will be tempted to add it back for optimization)
- Document in `__init__` that `_check_schema_version()` must run before `_check_weights_changed()`
- `_enrich_for_embedding()` method becomes dead code after noop enrichment — consider removing in this plan

#### Risk Assessment: **MEDIUM**

The three-layer cache correctness issue (CONCERN-01) is the primary risk — a subtle bug that only manifests when identical chunk text appears across files, which degrades search quality silently rather than failing loudly. The fix is straightforward but getting it wrong is dangerous.

### Plan 03 — Search Logging and Tests

#### Summary

Well-structured test plan that covers the key invariants. The canonical test directly validates the phase goal. The search_log feature is a clean, non-invasive addition. However, there is one missing test for the CONCERN-01 cache ordering fix.

#### Strengths

- Canonical invariant test (`metadata_only_reindex_exactly_one_tei_call`) directly validates the phase goal — 1 encode_batch call, 1 text for metadata-only changes
- Missing e_text fallback test covers edge case from Plan 02 Case 2
- Schema wipe test validates all 7 components are cleared
- Weight change test validates pure-math recompute (no TEI calls)
- search_log caps at 10,000 rows, DELETE with subquery ORDER BY id is correct for FIFO trim
- Non-fatal logging — search failures don't propagate to callers
- Logs parsed weights (not raw env var string) — consistent across weight changes
- `mode` + `reranked` columns added for future analytics

#### Concerns

- **[MEDIUM] CONCERN-06: No test for the three-layer cache correctness (CONCERN-01)**

  After fixing CONCERN-01, there should be a test that verifies `_embed_chunks()` returns e_text (not e_fused) when VecComponentStore has entries and vec_meta also has entries for the same text_hash. Without this test, a future developer could re-add the vec_meta lookup and reintroduce the bug.

  **Suggested test:** `test_embed_chunks_returns_etext_not_efused_from_cache` — pre-populate VecComponentStore with e_text and vec0 with a different e_fused for the same text_hash, verify `_embed_chunks()` returns e_text.

- **[LOW] CONCERN-07: search_log concurrent write with trickle**

  `log_search()` writes to `self._conn` (shared with trickle indexer). Both operate on `index.db` in WAL mode. The try/except handles failure. Acceptable but worth a comment.

- **[LOW] CONCERN-08: search_log trim query efficiency**

  `id INTEGER PRIMARY KEY AUTOINCREMENT` is the rowid and implicitly indexed. The explicit `CREATE INDEX on id` is redundant but harmless.

#### Suggestions

- Add the CONCERN-06 test to `test_pipeline_metadata.py`
- Add a test for empty frontmatter `meta_checksum` (title=None, tags=None) to `test_meta_checksum.py`
- Consider `DEFAULT CURRENT_TIMESTAMP` for `search_log.timestamp` instead of requiring caller to pass ISO string

#### Risk Assessment: **LOW**

The test plan is comprehensive and search_log is isolated. Primary gap is the missing test for CONCERN-01.
