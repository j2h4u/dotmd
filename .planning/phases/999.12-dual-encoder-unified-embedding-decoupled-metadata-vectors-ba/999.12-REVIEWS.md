## Codex Review

### Plan 01 — Foundation

**Summary**

Plan 01 is solid as a foundation plan. It defines the component storage boundary, introduces
metadata-only checksums, and makes weights explicit and validated. The main risk is that
canonicalization for `meta` entity IDs is documented but not enforced by the store itself,
leaving correctness dependent on every caller.

| HIGH-4 meta reads may not be normalized | PARTIALLY RESOLVED | Plan mandates canonical resolved path in docstring, but enforcement deferred to Plan 02 `_meta_entity_id()` usage |

---

### Strengths

- `VecComponentStore.store()` not committing is the right transaction boundary for pipeline-owned atomic writes
- Table key `(entity_id, component)` matches the dual-component model cleanly
- `meta_checksum()` correctly excludes body text and avoids O(N chunks) invalidation
- Weight validation requiring both `text` and `meta` prevents silent partial configs

---

### Concerns

- **MEDIUM: Canonical path normalization is only documented in the store.** If any caller passes a non-resolved path, metadata cache misses or duplicate meta rows can occur.
- **MEDIUM: `INSERT OR REPLACE` deletes then inserts in SQLite semantics.** Usually fine here, but if future triggers or foreign keys are added, it can have surprising behavior. `ON CONFLICT DO UPDATE` would be safer if SQLite version permits.
- **LOW: `meta_checksum()` excludes `kind` because kind changes hit `chunk_checksum`.** That coupling should be tested directly because it is easy to break later.
- **LOW: Weight parsing should reject unknown keys or duplicate keys,** not only require `text` and `meta`.

---

### Suggestions

- Add a private normalization helper in `VecComponentStore` only if the store is allowed to know component semantics; otherwise keep Plan 02's `_meta_entity_id()` but add tests proving raw relative paths are never used by pipeline call sites.
- Make `parsed_embedding_weights` reject extra keys like `text=0.7,meta=0.2,title=0.1`.
- Prefer `INSERT ... ON CONFLICT(entity_id, component) DO UPDATE SET embedding=excluded.embedding` if supported.

---

### Risk Assessment

**LOW to MEDIUM.** The foundational API is straightforward, but correctness depends on Plan 02
consistently applying canonical entity IDs.

### Plan 02 — Pipeline Refactor

**Summary**

Plan 02 addresses the Cycle 3 HIGHs substantially. The split between raw text components,
metadata components, and fused vec0 storage is now conceptually correct, and the cache ordering
fix is important. Remaining risk is mostly integration complexity: the plan names many
overlapping paths, and a few caller responsibilities are still implicit enough that
implementation could drift.

---

### Strengths

- Removing `lookup_embeddings_by_text_hash()` for post-999.12 text retrieval is correct. Reusing fused vectors as text vectors would cause compounding fusion errors.
- `_embed_chunks()` becoming pure is a good separation: retrieval/encoding only, writes owned elsewhere.
- Two-phase TEI-outside-transaction and writes-inside-transaction is the right direction for SQLite lock behavior.
- Schema version wipe recognizes the `text_hash` semantic break and avoids mixed old/new cache state.
- `weights_used` sentinel behavior avoids claiming success after partial recompute.
- Dimension mismatch raising `ValueError` is correct; silent truncation would corrupt ranking.

---

### Concerns

- **HIGH: `_embed_existing_chunks(model_switch=True)` routing remains underspecified.** The plan defines the parameter, but no caller in the plan is specified to set `model_switch=True`. If no caller sets it during model/schema changes, stale e_text could still be reused. The trigger condition for `model_switch=True` is missing.

- **HIGH: `_index_file_embed()` and `_save_and_embed_chunks()` may duplicate responsibilities.** Both appear to embed metadata, store text components, fuse, and add chunks depending on path. The plan needs a sharper ownership boundary: either `_index_file_embed()` is the single per-file transaction owner (with `_save_and_embed_chunks()` delegating to it), or bulk has its own equivalent path. Without this clarity, implementation could create double TEI calls.

- **MEDIUM: Bulk `run()` says it batches all e_meta across files, then "per-file: embed chunks → store e_text → fuse → add_chunks."** It does not explicitly state whether each file has its own transaction or one large transaction. With several days of rebuild time, one giant transaction is risky; per-file or bounded batch transactions are safer.

- **MEDIUM: `_check_schema_version()` wipes 7 stores in one transaction.** If vec0 virtual table operations have special transaction behavior under sqlite-vec, this needs verification.

- **MEDIUM: Metadata-only fast path falls back to full embed if e_text BLOBs are missing.** Correct behavior, but it should also repair missing component rows inside the same transaction after fallback.

- **MEDIUM: `_meta_entity_id()` as a static method on pipeline may not protect other modules** from using raw paths when accessing `VecComponentStore`.

- **LOW: Recomputing weights in batches of 1000 files could still be expensive** on 16GB RAM depending on vector dimensions and chunk fanout.

---

### Suggestions

- Explicitly define the model-switch caller. For example: model-name change or vector model suffix mismatch routes existing chunk rebuild through `_embed_existing_chunks(..., model_switch=True)`.
- State the exact ownership boundary: `_embed_chunks()` returns e_text only. `_embed_meta_component()` returns e_meta only. `_index_file_embed()` is the ONLY method that writes vec components, fused vec0, trackers, and commits for one file. Bulk batching may precompute e_meta, but should pass it into the same transaction-writing path.
- Add an explicit commit rule for bulk: commit per file or per bounded batch.
- List the 7 `_meta_entity_id()` call sites directly in the implementation plan, not only in a docstring.
- Add tests where relative and absolute paths refer to the same file and must hit the same metadata vector.
- Add failure tests: TEI succeeds but DB write fails; verify no partial tracker advance.

---

### Risk Assessment

**MEDIUM.** The design is mostly correct, but the implementation has enough moving parts that
unclear routing and transaction ownership could reintroduce stale vector reuse or partial
indexing bugs.

### Strengths

- Polymorphic entity_id key space in a single table is clean — no need for separate tables per component
- `store()` not committing keeps transaction control with callers
- `delete_by_entity_ids()` returning row count allows callers to detect unexpected deletes
- `count()` returning 0 on exception prevents startup failures from empty-store guard bugs

### Concerns

- **LOW: `entity_id` key space collision undocumented.** The module docstring mandates key convention (`chunk_id` for text, resolved path for meta) but doesn't explicitly state these two namespaces must never overlap. If a `chunk_id` happens to equal a canonical file path string, the wrong component could be returned. Unlikely in practice but should be documented as an invariant.

- **LOW: `meta_checksum(None, None)` edge case.** Plan says "None-safe" but doesn't specify what `meta_checksum` returns when both title and tags are absent. If it returns a fixed hash (e.g., hash of empty string), then every file with no frontmatter gets the same `meta_checksum`, meaning any frontmatter-only change to one such file would not be detected for others. Should return `None` (skip meta embedding) for genuinely empty metadata, and callers should handle `None` → skip meta fusion.

- Document in `VecComponentStore` module docstring that `entity_id` is polymorphic and the two key spaces must never overlap
- Specify `meta_checksum` return value for `(None, None)` — recommend returning `None` and having the pipeline skip meta embedding (use `e_fused = normalize(e_text)` with `w_text=1.0`)

---

### Risk Assessment

**LOW** — Clean foundation with well-defined interfaces. Minor edge cases around empty metadata.

### Plan 02 — Pipeline Refactor

**Summary**

The critical plan of the phase. Addresses all 4 Cycle 3 HIGHs with specific method signatures,
call chains, and transaction boundaries. The 3-layer embedding cache replacing
`lookup_embeddings_by_text_hash()` is the most important correctness fix — it prevents
double-fusion by reading `e_text` directly from `VecComponentStore` instead of reading
`e_fused` from `vec0`. Transaction boundary design (TEI outside BEGIN, writes inside) is sound.
However, there are two specification gaps in the caller/callee contracts that could cause
integration bugs.

---

### Prior-Cycle HIGH Verification

| Cycle 3 HIGH | Verdict | Evidence |
|---|---|---|
| HIGH-1: Bulk path chunk-to-file mapping | **RESOLVED** | `_save_and_embed_chunks()` now explicitly accepts `file_info`; M2M last-write-wins documented; `test_bulk_fusion_pairing.py` covers this case |
| HIGH-2: `_embed_existing_chunks()` conflation | **RESOLVED** | Split into two sub-paths via `model_switch: bool` parameter with distinct behavior per path |
| HIGH-3: `_index_file_embed()` call sites | **RESOLVED** | Explicit call chain diagram: trickle path, bulk path (two entry points), with `_index_file_embed` as terminal transaction owner |
| HIGH-4: Meta component path normalization | **RESOLVED** | `_meta_entity_id()` static method centralizes `str(pathlib.Path(path).resolve())`; 7 call sites enumerated in docstring |

---

### Strengths

- Critical correctness fix: removing `lookup_embeddings_by_text_hash()` (which JOINs vec_meta with vec0 containing e_fused) and replacing with `VecComponentStore.get_batch(chunk_ids, "text")` eliminates the double-fusion bug
- 3-layer cache (VecComponentStore → embedding_cache → TEI) correctly handles M2M shared chunks (same text_hash, different chunk_ids)
- Two-phase transaction design (TEI outside BEGIN, writes inside) prevents holding locks during network calls
- Schema version wipe: single transaction wrapping all 7 wipes, sentinel written LAST — correct crash-safety semantics
- `_check_weights_changed()` guard: `vec_components.count() == 0 → return` prevents infinite loop after schema wipe
- Partial recompute safety: `files_skipped > 0` → don't update sentinel → self-healing on next startup
- `_fuse_vectors()` raises `ValueError` on dimension mismatch — never truncates silently
- `_embed_chunks()` is PURE — no storage writes, clean separation of concerns

---

### Concerns

- **MEDIUM: `_save_and_embed_chunks()` vs `_index_file_embed()` TEI responsibility overlap.** Point 8 says `_save_and_embed_chunks` "calls `_embed_meta_component(file_info)` for e_meta." Point 7 says `_index_file_embed` is the "transaction owner" with "Case 1 (body_changed=True): full path — encode e_text + e_meta." If both methods make TEI calls, the contract is ambiguous: does `_save_and_embed_chunks` pre-compute e_text AND e_meta, then pass them to `_index_file_embed`? Or does `_save_and_embed_chunks` only save chunks to DB, then `_index_file_embed` does all TEI + writes? The current wording suggests both do TEI work, which would mean double TEI calls for the bulk path. The plan should specify which method owns TEI calls.

- **MEDIUM: `model_switch=True` caller unspecified.** Point 9 defines `_embed_existing_chunks(file_info, chunks, *, model_switch=False)` with two distinct sub-paths, but no caller in the plan sets `model_switch=True`. The bulk `run()` path (point 10) doesn't mention it. Schema version wipe (point 12) empties all tables. Table names include model suffix, so a model config change produces new empty tables — meaning existing chunks wouldn't exist in the new table. The `model_switch=True` path may be dead code, or the trigger condition is missing from the plan.

- **LOW: `weights_used` sentinel not set during normal indexing.** After schema wipe + initial bulk index, `vec_components` is populated but `weights_used` sentinel is never written (normal indexing path doesn't set it; `_check_weights_changed()` only sets it after a successful full recompute). On the next startup, `_check_weights_changed()` sees `count > 0` with no sentinel → triggers a full recompute of all fused vectors — redundant work. Should specify that the sentinel is written once after the initial bulk index completes, or that `_check_weights_changed()` handles the "no sentinel exists" case by writing (not recomputing) if the current weights match.

- **LOW: `_embed_existing_chunks` in bulk `run()` branching condition not documented.** The plan implies `_save_and_embed_chunks` for body_changed=True files and `_embed_existing_chunks` for metadata_changed=True only, but doesn't state this explicitly.

---

### Suggestions

- Clarify TEI ownership for the `_save_and_embed_chunks → _index_file_embed` chain. Recommended: `_index_file_embed` ALWAYS owns TEI calls + transaction. `_save_and_embed_chunks` ONLY persists chunk text/FTS data to DB, then delegates embedding to `_index_file_embed(file_info, chunks, body_changed=True, metadata_changed=True)`.
- Either specify when `model_switch=True` is set (e.g., "set when `DOTMD_EMBEDDING_MODEL` config changes between runs and table suffixes match due to model aliasing") or remove the parameter and document that model changes always result in new tables (empty, no existing chunks to re-embed).
- Add a `weights_used` sentinel write at the end of `_check_schema_version()` (write current weights when schema version matches) or at the end of the initial bulk `run()` to prevent the redundant second-startup recompute.

---

### Risk Assessment

**MEDIUM** — The two MEDIUM concerns are specification gaps in critical paths (TEI call
ownership and model_switch routing). Neither is a correctness bug in the plan as written, but
both could cause integration bugs during implementation if the ambiguity is resolved incorrectly.
The Cycle 3 HIGHs are genuinely resolved.

### New Cycle 4 Concerns

**[MEDIUM — Both Reviewers] TEI ownership between `_save_and_embed_chunks()` and `_index_file_embed()` is ambiguous**

Plan 02 says `_save_and_embed_chunks()` calls `_embed_meta_component(file_info)` for e_meta
(implying TEI inside that method), AND that `_index_file_embed()` is the transaction owner
with "Case 1 (body_changed=True): full path — encode e_text + e_meta" (implying TEI inside
`_index_file_embed`). If both methods make TEI calls, the bulk path could produce double TEI
calls. The fix: either `_index_file_embed` is the sole TEI caller (and `_save_and_embed_chunks`
delegates to it), or `_save_and_embed_chunks` pre-computes vectors and passes them to
`_index_file_embed` without further TEI work.

**[MEDIUM — Both Reviewers] `model_switch=True` caller is not specified anywhere in the plan**

`_embed_existing_chunks(model_switch=True)` sub-path exists in the plan, but no call site sets
`model_switch=True`. Since model name changes produce new table suffixes (new empty tables),
existing chunks don't exist in the new table — making `model_switch=True` potentially dead
code for this codebase's table-naming scheme. The plan should either specify when the flag is
set (with a concrete trigger condition) or remove it and document that model changes produce
fresh tables.

**[LOW — OpenCode] `weights_used` sentinel not written after initial bulk index**

After a schema wipe and initial full index, `vec_components` is populated but `weights_used`
in `vec_config` is never written. On the next startup, `_check_weights_changed()` sees
`count > 0` with no stored sentinel → triggers a full recompute of all fused vectors
(redundant, since the current weights were already used). The plan should write the sentinel
at the end of the initial bulk index or add a "no sentinel present → write sentinel without
recomputing" branch to `_check_weights_changed()`.

3. **Write `weights_used` sentinel after initial bulk index** (LOW, OpenCode)

4. **Add TEI-call-count test for `_save_and_embed_chunks → _index_file_embed`** (MEDIUM, Plan 03)
   — Regression protection for double-TEI-call bug in bulk path.
