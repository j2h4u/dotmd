## Codex Review

### Summary

The Cycle 3 plans are materially stronger than the previous version. The main architectural
correction is now explicit: stored `vec0` vectors are fused retrieval vectors, while
`VecComponentStore` is the authoritative source for raw component vectors. That separation
addresses the original cache-ordering bug and makes the metadata-only fast path plausible. The
plan is still medium risk because it changes ingestion, reindexing, schema invalidation,
fingerprint semantics, cache semantics, and search logging in one phase. The biggest remaining
risks are transactional consistency across M2M chunk/file relations, stale or missing component
cleanup, and ensuring every vec0-writing path uses fused vectors with identical entity-id
conventions.

### Strengths

- Clear separation between `e_text`, `e_meta`, and `e_fused`; matches desired production behavior
- `VecComponentStore` has a simple, durable schema with composite PK and caller-owned transactions
- Removing enriched text from embedding paths attacks the actual root cause instead of adding cache workarounds
- Weight changes handled locally without TEI calls, with batching and partial-recompute retry semantics
- Schema wipe is atomic — sentinel written last inside single BEGIN...COMMIT
- Tests cover important regressions: metadata-only TEI count, cache ordering, checksum orthogonality, missing component fallback, weight recompute
- Keeping `e_fused` in existing `vec0` avoids a broad search-path rewrite

---

### Concerns

- **HIGH: Meta component reads may not be normalized everywhere.**
  `_embed_meta_component()` stores canonical paths, but paths used in metadata-only reindex,
  `reindex_vectors()`, `_check_weights_changed()`, and `_embed_existing_chunks()` must also
  canonicalize before `get(..., "meta")`. A single non-canonical read silently misses stored
  `e_meta` and falls back to full re-embed.

- **HIGH: Stale text components may survive M2M unlink/delete cases.**
  With content-addressed chunks and `chunk_file_paths_*`, deleting by file path is wrong if
  another file still references the same chunk. Deleting by chunk_id too aggressively can break
  shared chunks. The plan needs explicit cleanup semantics: only delete `text` components when
  the chunk row itself is removed or has no remaining file-path links.

- **HIGH: Transaction boundary may not cover all related writes.**
  `_index_file_embed()` owns embed→fuse→store, but TEI calls should happen outside the SQLite
  write transaction. The durable write phase should atomically update metadata, FTS/path links,
  vec0, fingerprints, and component rows. Holding a transaction across TEI calls risks lock
  contention and failed long transactions.

- **MEDIUM: Bulk `run()` batch e_meta encoding conflicts with per-file transaction ownership.**
  Plan 02 batches all `e_meta` in a single `encode_batch`, while `_index_file_embed()` owns
  transactions. The plan should define where precomputed meta vectors are passed, stored, and
  retried if only some files fail.

- **MEDIUM: Fallback for missing `e_text` BLOBs can violate the "1 TEI call" metric.**
  Intended behavior, but production observability should distinguish steady-state fast path from
  degraded fallback — otherwise a missing component bug looks like normal metadata churn.

- **MEDIUM: `EmbeddingCache` semantics after 999.12 need a hard boundary.**
  Layer 2 (`embedding_cache.lookup(text_hashes)`) is only safe if the cache stores raw text
  embeddings (not fused vectors) under chunk text hashes. Existing rows wiped on schema bump,
  but future writes must not store fused vectors under chunk text hashes.

- **MEDIUM: Search logging write hotspot on every insert.**
  Trimming with repeated count/delete on every insert can be expensive. Should use bounded
  DELETE by id after insert.

- **LOW: `search_log_id_idx` is redundant.** `id INTEGER PRIMARY KEY` already creates rowid access.

- **LOW: `meta_checksum` tag handling may be too narrow.** `"a,b"` vs `["a", "b"]` frontmatter
  formats would produce different checksums. May be acceptable but should be intentional.

- **LOW: Weight validation float tolerance.** `math.isclose(sum, 1.0, abs_tol=1e-6)` is more
  robust than exact `abs(total - 1.0) > 0.001` for decimal representation edge cases.

---

### Suggestions

- Centralize canonical entity-id generation in a `_meta_entity_id(path)` helper; use it in all
  store/get call sites for `meta` component to prevent normalization divergence.
- Specify component lifecycle rules for shared chunks: `text` component deleted only when chunk
  has no remaining file-path links; `meta` component deleted when its file is removed.
- Make `_index_file_embed()` two-phase: (1) compute TEI + fused vectors outside transaction,
  (2) atomically write all state inside transaction.
- Add tests for every vec0-writing path: trickle metadata-only, trickle body-change, bulk
  `_save_and_embed_chunks()`, `_embed_existing_chunks()`, `reindex_vectors()`,
  `_check_weights_changed()`.
- Add one test proving `vec0` receives `e_fused` while `vec_components(..., "text")` stores raw
  `e_text`.
- Add structured logging counters for fast-path files, fallback files, TEI calls by reason.

---

### Risk Assessment

**Overall risk: MEDIUM.** Plans resolve prior high-severity design flaws and include a good
regression-test suite. Remaining risk comes from implementation breadth: multiple ingestion paths
must be updated consistently, and silent production failures are possible if entity IDs,
component lifecycle, or vec0 write semantics diverge in one path.

### Summary

The three plans form a well-structured wave-based implementation that correctly addresses the
root cause. The core design (e_text cached per chunk + e_meta per file + local fusion) is sound
and all prior Cycle 2 HIGHs are properly resolved. The primary remaining risks are at the
integration boundary between the new per-file `_index_file_embed()` and the existing
batch-oriented `_save_and_embed_chunks()` / `_ingest_and_finalize()` call chains, where the
plan descriptions don't fully specify how chunks get mapped back to their source files for e_meta
pairing and fusion in the bulk path. Additionally, `_embed_existing_chunks()` must handle two
semantically different cases (model-switch vs. metadata-only) that the plan conflates into one
description.

---

### Strengths

- **Clean wave separation:** Plan 01 creates new code without modifying existing code — independently testable and risk-free
- **Pure `_embed_chunks()` refactor:** Making this method storage-free (returns e_text only, caller owns writes) is a clean separation of concerns
- **VecComponentStore design:** INSERT OR REPLACE, no auto-commit, batch-500 — all match existing patterns in `EmbeddingCache` (cache.py:117-156)
- **Fusion math uses stdlib only:** Critical for AVX2-constrained hardware
- **Schema wipe atomicity:** Single BEGIN...COMMIT wrapping all 7 component clears prevents partial state
- **`_check_weights_changed()` guard on empty store:** Prevents OOM on fresh install; sentinel-not-updated-on-partial-recompute is correct
- **Test coverage is comprehensive:** 35 tests across 5 files covering component storage, fusion math, checksums, pipeline flows, and the critical regression case

---

### Concerns

- **HIGH: Bulk path chunk-to-file mapping for e_meta fusion is underspecified.**
  The current `_save_and_embed_chunks()` (line 772) processes ALL chunks in one batch. After
  the refactor, e_text is computed in batch but e_meta must be paired per-file for fusion. The
  plan states "bulk run() path: batch e_meta encoding across all files" but doesn't specify:
  (1) how chunks are grouped back to source files (via `chunk_file_paths` M2M? via
  `chunk.file_paths[0]`?), (2) where in `_save_and_embed_chunks()` the e_meta batch encode +
  per-chunk fusion happens, (3) whether `_save_and_embed_chunks()` continues as the batch entry
  point or whether `_index_file_embed()` replaces it per-file. The current flow is
  `_save_and_embed_chunks()` → `_embed_chunks()` → `add_chunks()`; after refactor it needs
  `_embed_chunks()` → group-by-file → encode e_meta per file → fuse per chunk → `add_chunks()`
  with e_fused. This is a significant structural change that the plan doesn't detail.

- **HIGH: `_embed_existing_chunks()` conflates model-switch and metadata-only cases.**
  The current `_embed_existing_chunks()` (line 1576) handles embed-only files. After the
  refactor, it must handle two distinct cases: (1) **model switch** — ALL embeddings
  invalidated, both e_text and e_meta must be re-encoded via TEI; (2) **metadata-only change**
  — e_text cached in VecComponentStore, only e_meta needs encoding (1 TEI call). Plan 02 states
  `_embed_existing_chunks()` "must compute e_meta + fuse + write all 3 components" but doesn't
  distinguish these cases. The risk: model-switch path may silently skip e_text re-encoding if
  the code assumes VecComponentStore always has valid e_text.

- **HIGH: `_index_file_embed()` call site is undefined.**
  Plan 02 introduces `_index_file_embed(file_info, chunks, *, body_changed, metadata_changed)`
  as the transaction owner, but doesn't specify where it's called from. The current pipeline has
  two embedding paths: `_save_and_embed_chunks()` (bulk, new/modified files from
  `_ingest_and_finalize()` line 726) and `_embed_existing_chunks()` (embed-only, from line 739).
  Does `_index_file_embed()` replace both? Is it called per-file inside a loop within
  `_save_and_embed_chunks()`? The plan must specify the exact call chain.

- **MEDIUM: M2M shared-chunk edge case undocumented.**
  When the same chunk text appears in File A (title="X") and File B (title="Y"), the chunk gets
  one `chunk_id` and one row in vec0. e_text is identical for both files, but e_meta differs.
  `_check_weights_changed()` grouping by file may re-fuse a shared chunk multiple times with
  different e_meta values — last write wins to vec0. This is a pre-existing design constraint,
  not a regression, but should be documented as known behavior.

- **MEDIUM: index() diff logic → meta_tracker mapping underspecified.**
  The current `_incremental_index()` (line 647) separates changed files from embed-only files.
  After the refactor, it needs per-file `body_changed`/`metadata_changed` flags from
  chunk_tracker and meta_tracker diffs. The plan describes `_index_file_embed()` flags but
  doesn't specify how the index-level diff produces them.

- **MEDIUM: search_log non-fatal trim may cause unbounded growth.**
  If the trim DELETE fails (lock contention during heavy indexing), rows accumulate beyond
  10,000. Consider a trim at startup or every N searches.

- **MEDIUM: trickle startup after schema wipe — operator expectation.**
  When `_check_schema_version()` wipes, all files appear as "new" (no fingerprints). First
  trickle cycle after deploy is a full rebuild. Should be documented in deployment notes so
  operators expect elevated TEI load.

- **LOW: search_log `mode` column — enum serialization.**
  `DotMDService.search()` accepts `SearchMode | str`. Ensure the enum `.value` is stored, not
  the repr.

- **LOW: `_fuse_vectors()` stdlib math precision note.**
  `math.sqrt(sum(x**2 for x in v))` vs numpy precision. Negligible for cosine similarity but
  worth a comment.

---

### Suggestions

1. **Specify bulk-path fusion pairing**: Add to Plan 02 Task 4: after `_embed_chunks()` returns
   e_text vectors, group chunks by `chunk.file_paths[0]` (or `chunk_file_paths` M2M), batch-encode
   e_meta per unique file, fuse per chunk, pass e_fused to `add_chunks()`. Resolves HIGH-1.

2. **Split `_embed_existing_chunks()`**: Two sub-paths — `_metadata_only_reembed()` (read cached
   e_text + encode e_meta, 1 TEI call per file) and `_model_switch_reembed()` (full re-encoding).
   Route based on trigger reason. Resolves HIGH-2.

3. **Document `_index_file_embed()` call chain**: Show exact method call graph from `index()` →
   `_incremental_index()` → `_ingest_and_finalize()` → where `_index_file_embed()` fits.
   Resolves HIGH-3.

4. **Document M2M shared-chunk fusion behavior**: Comment that for shared chunks,
   `file_paths[0]`'s e_meta is used (matches current enrichment behavior). Resolves MEDIUM-1.

5. **Add test for bulk-path chunk-to-file fusion pairing**: Multiple files whose chunks must be
   correctly paired with per-file e_meta in `_save_and_embed_chunks()`. Would catch HIGH-1
   class bugs.

6. **Add test for M2M shared-chunk**: Two files with same body, different titles → verify chunk
   gets fused with primary file's e_meta, e_text identical for both.

---

### Risk Assessment

**Overall risk: MEDIUM.** Plans are well-designed and all prior HIGHs are resolved. Remaining
risk concentrated in Plan 02's integration with existing pipeline. Three HIGHs are specification
gaps (not design flaws) — the architecture is correct, but the implementation path needs more
detail around bulk fusion pairing, model-switch routing, and call chain specification.

**Recommendation:** Address the three HIGHs with explicit call-chain documentation and
file-to-chunk grouping logic before execution. MEDIUMs can be addressed during implementation.

### New Cycle 3 HIGHs

**[HIGH] Bulk path chunk-to-file mapping for e_meta fusion is underspecified (OpenCode HIGH-1)**

Both reviewers flagged this in different forms. OpenCode identified the concrete specification
gap: `_save_and_embed_chunks()` processes all chunks in one batch, but e_meta must be paired
per-file for fusion. Codex identified the same concern via "bulk run() batch e_meta encoding
conflicts with per-file transaction ownership." The fix is to explicitly specify how chunks are
grouped back to source files and where in the bulk path per-file fusion happens.

**[HIGH] `_embed_existing_chunks()` conflates model-switch and metadata-only cases (OpenCode HIGH-2)**

OpenCode identified this precisely: the method at line 1576 handles embed-only files but must
now distinguish model-switch (full re-encode both e_text and e_meta via TEI) from metadata-only
(read cached e_text + 1 TEI call for e_meta). Without this distinction, a model-switch may
silently use stale e_text from VecComponentStore. Codex touched this via "meta component reads
may not be normalized everywhere" and "stale text components may survive M2M cases."

**[HIGH] `_index_file_embed()` call site is undefined (OpenCode HIGH-3)**

The plan introduces `_index_file_embed()` as the transaction owner but doesn't specify where it
is called from relative to `_save_and_embed_chunks()` and `_embed_existing_chunks()`. Without
this, the implementation may create ambiguous double-writes or miss the transaction boundary.

**[HIGH] Meta component reads may not be normalized everywhere (Codex HIGH-1)**

Codex raised that while `_embed_meta_component()` stores canonical paths, every GET call site
for `meta` component (metadata-only reindex, `reindex_vectors()`, `_check_weights_changed()`,
`_embed_existing_chunks()`) must also canonicalize. A single non-canonical read silently falls
back to full re-embed. OpenCode's suggestion to create a `_meta_entity_id(path)` helper
addresses this.
