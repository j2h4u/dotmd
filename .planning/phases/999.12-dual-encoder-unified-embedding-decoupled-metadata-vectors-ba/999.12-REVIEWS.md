# Cross-AI Plan Review — Phase 999.12

## Codex Review

## Summary

The three-plan sequence is coherent and mostly aligned with the phase goal: separating stable chunk text embeddings from mutable document metadata embeddings, then locally recomputing fused vectors to avoid N TEI calls on metadata-only changes. The decomposition is sensible: Plan 01 establishes primitives, Plan 02 changes the indexing semantics, and Plan 03 verifies the operational invariant. The main risks are in Plan 02: cache invalidation, tracker semantics, vec0 replacement correctness, transaction boundaries, and ensuring every path distinguishes body changes from metadata-only changes without accidentally skipping required updates.

## Strengths

- Clear separation between `e_text`, `e_meta`, and `e_fused`.
- Keeps ANN search unchanged by continuing to store fused vectors in existing `vec0`.
- Good decision to store component vectors as BLOBs and use local stdlib math, given CPU and AVX2 constraints.
- Metadata-only fast path directly targets the failure mode: recompute one document-level metadata vector, then locally update all chunk fused vectors.
- Wipe-on-deploy is pragmatic for incompatible `text_hash` semantics.
- Weight config is planned early, which avoids hardcoding assumptions into fusion code.
- Tests explicitly target the most important invariant: metadata-only changes should cause exactly one TEI call.

## Concerns

### Plan 01

- **MEDIUM:** `meta_checksum(path)` probably needs to operate on parsed file metadata, not just `path`, unless `reader.py` already resolves title/tags internally from path. The plan should specify the function signature and data source clearly.
- **MEDIUM:** The `tags_str` normalization needs to handle absent tags, scalar tags, list tags, duplicate tags, mixed case, and YAML ordering consistently.
- **LOW:** `VecComponentStore.store()` doing no commit is consistent with existing store patterns only if every caller is already transaction-aware. This should be documented and tested with the actual pipeline commit flow.
- **LOW:** Table names derived from strategy/model need sanitization or reuse of the existing vector store naming logic. Model names often contain `/`, `-`, `.`, or `:`.

### Plan 02

- **HIGH:** Replacing `embed_tracker` with `meta_tracker` is under-specified. The system still needs to know independently whether body/chunks changed and whether metadata changed. The stated decision says `chunk_tracker = hash(body+kind)` unchanged and `meta_tracker = hash(title+tags)`, but the plan mostly describes a swap. Avoid losing the body-change detection path.
- **HIGH:** Metadata-only fast path must prove that all current chunks still exist, all chunk IDs are stable, and all required `e_text` components are present. If chunk IDs are regenerated on metadata changes or depend on enriched text, the fast path may miss.
- **HIGH:** `_embed_chunks()` adding a third cache layer from `VecComponentStore` needs strict validation against `text_hash`. A stored `e_text` BLOB keyed only by `chunk_id` is dangerous if chunk IDs can collide or remain stable while body text changes. The component store lookup should be guarded by current `vec_meta.text_hash` or another body checksum.
- **HIGH:** Wipe-on-deploy clearing "both trackers" must include all relevant persistent state: vec0, vec_meta, embedding cache, component table, chunk tracker, meta tracker, and schema sentinel. Missing one can produce false cache hits with incompatible vectors.
- **MEDIUM:** `content_handlers.py` changing all embedding enrichers to noop may affect any non-vector consumer that reused the same registry. The plan notes FTS5 still needs enrichment, but it should identify the exact FTS5 path to ensure it does not use the modified registry.
- **MEDIUM:** `reindex_vectors()` grouping by file could be tricky if chunks lack reliable file metadata or if duplicate file paths exist after moves/renames.
- **MEDIUM:** Weight changes need a dedicated recompute path. D-09 says changing weights is cheap and should recompute `e_fused`, but Plan 02 does not clearly say where weight-change detection happens or how stale fused vectors are invalidated.
- **MEDIUM:** Fusion dimension mismatch handling is not mentioned. `e_text` and `e_meta` must have identical dimensions; corrupted BLOBs or model changes should fail clearly and trigger rebuild.
- **MEDIUM:** Local fusion for every metadata update still performs N DELETE+INSERT updates into vec0. That is acceptable for ~8 chunks/file, but the original incident mentioned ~107 chunks. It is still far cheaper than TEI, but should be batched in one transaction.
- **LOW:** Logging "1 TEI call" is useful but weak as an acceptance criterion. Tests should assert encode call count, not rely on logs.

### Plan 03

- **MEDIUM:** `search_log` may add write contention to `index.db`, especially if search can run concurrently with indexing. Non-fatal try/except helps, but busy timeouts and transaction behavior should be considered.
- **MEDIUM:** Retention via `DELETE oldest on overflow` can become inefficient if done every search with `COUNT(*)`. Prefer deleting only when needed and using indexed `id`/`timestamp`.
- **MEDIUM:** `weights_used` should log the parsed effective weights, not the raw env var. This matters for future calibration/debugging.
- **MEDIUM:** `top_results JSON` should avoid logging full chunk text if privacy or DB growth matters. Store stable identifiers, scores, and maybe short metadata.
- **LOW:** Tests should include invalid/corrupt component BLOB handling, dimension mismatch, and missing `e_text` fallback behavior.
- **LOW:** `log_search()` should be tested for failure isolation by forcing an insert error and confirming `DotMDService.search()` still returns results.

## Suggestions

- Explicitly model three independent state dimensions:
  - body/chunk state via `chunk_tracker`
  - metadata state via `meta_tracker`
  - fusion config state via stored `weights_used` or `weights_checksum`
- Add a `fusion_config` or `vec_config` entry for effective weights. On startup, if weights changed, recompute `e_fused` from components without TEI.
- Guard component cache hits with body hash validation: use `chunk_id + text_hash`, or store `checksum` alongside each component, or only trust `e_text` if `vec_meta.text_hash == current_body_text_hash`.
- Make chunk ID stability an explicit invariant. If chunk IDs include text hash, metadata-only updates are fine; if they include enriched text, this phase must change that.
- Add transactional boundaries to Plan 02: compute/store `e_meta` → compute/store all missing `e_text` → compute fused vectors → update vec0/vec_meta → update trackers → commit once.
- Add tests for: metadata-only update with multiple chunks (exactly one metadata encode, zero text encode); body-only update; metadata + body update; missing one `e_text` BLOB; changed weights (no TEI calls); model/schema dimension mismatch.
- Keep `enrich_with_title_and_tags` naming clear. If it remains used for FTS5 only, consider renaming as `fts_enrich_with_title_and_tags` in a later cleanup.

## Risk Assessment

**Overall risk: MEDIUM-HIGH.**

The architecture is sound and directly addresses the phase goal, but Plan 02 changes core indexing invariants: hashing semantics, tracker behavior, cache lookup order, vector storage, and all embedding call sites. The highest-risk area is stale-vector correctness: an incorrect cache hit could silently preserve the wrong `e_text`, while an incorrect tracker update could skip necessary recomputation. With stronger checksum guards, explicit weight-change invalidation, transactional updates, and the additional tests above, the implementation risk drops to medium.

## OpenCode Review

## Plan 01 — Foundation: VecComponentStore, meta_checksum, weights config

### Summary

Plan 01 creates three clean, well-scoped primitives that are independently testable and have zero impact on existing functionality. The design mirrors existing patterns (`SQLiteVecVectorStore`, `embed_checksum`, Pydantic validators) which reduces cognitive overhead and integration risk. This is a solid foundation layer.

### Strengths

- **Pattern consistency**: VecComponentStore mirrors `SQLiteVecVectorStore` conventions (batched 500-group queries, `struct.pack` float32, caller-owns-commit) — the team already knows this pattern.
- **Pure stdlib constraint**: No numpy dependency respects the AVX2 hardware limitation. Good.
- `meta_checksum` placement after `embed_checksum` in `reader.py` gives a clean before/after comparison for review.
- Pydantic `field_validator` for weight sum is the right place — fails at startup, not at search time.

### Concerns

- **MEDIUM**: The table name `vec_components_<strategy>_<model>` isn't explicitly specified as matching the `sqlite_vec.py` naming convention. The current convention is `vec_chunks_{strategy}{model_suffix}` (no underscore between strategy and model). Plan 01 uses underscores on both sides. This will cause confusion if not aligned. The plan should specify the exact naming function and confirm it's consistent with `_model_to_table_suffix()`.
- **LOW**: `meta_checksum` uses `blake3.blake3(f"{title}\n{tags_str}".encode())` but doesn't include `kind` in the hash. The current `embed_checksum` includes `kind`. If a file's `kind` changes (e.g. document → meeting_transcript), `meta_checksum` won't detect it. Since `kind` affects chunking via `pre_split`, `chunk_checksum` will catch it — but `meta_checksum` won't. If kind changes, the pre_split changes, body gets re-chunked, and the entire pipeline runs anyway (Phase 1 triggers), so e_meta gets recomputed as part of the full path. This is fine, but should be documented as a design decision.
- **LOW**: `VecComponentStore` constructor signature isn't specified. Will it take `conn` like the existing stores? Will it share the same unified connection? The plan says "mirroring SQLiteVecVectorStore patterns" but that class accepts both `db_path` and `conn`. Clarity needed.

### Suggestions

- Add a naming helper function (like `_model_to_table_suffix`) for the components table name, or explicitly state it reuses the same function.
- Document the `kind` exclusion from `meta_checksum` as intentional with rationale.

### Risk Assessment: **LOW**

Three isolated primitives with no existing code mutations. Tests are straightforward. The worst case is a naming convention mismatch caught in Plan 02 integration.

## Plan 02 — Pipeline Refactor: embed_tracker → meta_tracker, e_text storage, metadata-only fast path

### Summary

Plan 02 is the core of the phase and carries the highest risk. It refactors the heart of `pipeline.py` — the embed path, fingerprinting, and caching logic — while introducing the dual-encoder fusion math. The five tasks are logically sequenced, but the plan has several integration gaps around the metadata-only fast path detection logic, transaction boundaries, and the interaction between `e_text` BLOB reads and the existing caching layers. The wipe-on-deploy strategy is pragmatic given the hardware constraints.

### Strengths

- **Wipe-on-deploy via schema_version** is the right call. Attempting a migration of existing vectors to the new component scheme would be complex and error-prone for 532 chunks. Rebuild is acceptable.
- **Three-layer cache** (vec_meta text_hash → global embedding_cache → VecComponentStore) is well-ordered: cheapest/most-targeted check first.
- **metadata-only fast path** is the core value prop — reading stored `e_text` BLOBs + 1 TEI call + local fusion. This correctly addresses the stated problem (tag sync → hours of re-indexing).
- General fusion formula `normalize(Σᵢ wᵢ · normalize(eᵢ))` is extensible to future components without re-architecture.
- Deprecation path for `enrich_with_title_and_tags` (kept for FTS5, removed from embedding) avoids a flag day.

### Concerns

- **HIGH**: **How does the metadata-only fast path get triggered?** The plan describes the fast path in `index_file()` but doesn't explain the detection logic. The pipeline needs to distinguish: (1) body changed (chunk_tracker fired) → full re-chunk + re-embed e_text + re-embed e_meta + fuse; (2) metadata only (meta_tracker fired, chunk_tracker didn't) → read e_text BLOBs + 1 TEI e_meta + fuse; (3) nothing changed → skip. The plan says "metadata-only fast path (index_file): reads stored e_text BLOBs" but doesn't show the branching logic. What if some e_text BLOBs are missing? The plan mentions "fallback if e_text BLOBs missing" but doesn't specify the fallback behavior.

- **HIGH**: **Transaction boundary for metadata-only fast path.** The fast path does: read BLOBs → 1 TEI call → local fusion → update vec0. This involves reading from VecComponentStore (SELECT), writing to vec0 (DELETE + INSERT via `add_chunks`), writing e_meta to VecComponentStore (INSERT OR REPLACE), updating meta_tracker fingerprint, possibly updating FTS5 and graph metadata. Are all of these in one transaction? If the TEI call succeeds but vec0 write fails, you have inconsistent state. The plan needs to specify the transaction boundary clearly.

- **HIGH**: **`_embed_chunks()` now stores to VecComponentStore mid-function.** Currently, `_embed_chunks()` is a pure "compute embeddings" function — it returns data, doesn't write to storage. The refactor has it writing e_text to VecComponentStore as a side effect. This breaks the function's current contract. Option 1: Keep `_embed_chunks()` pure (return e_text vectors) and have the caller store them. Option 2: Rename to `_embed_and_store_text_components()` to signal the side effect. Option 1 is cleaner for testability and aligns with the existing pattern.

- **MEDIUM**: **text_hash semantics change affects embedding_cache.** The `embedding_cache` is shared across all strategies/models. If there are other strategy tables that weren't wiped (the plan only wipes vec_components, vec0, vec_meta, trackers for the current strategy+model), their cache entries will have stale `text_hash` values. If `embedding_cache` is wiped entirely (not just per-strategy), this is fine. Confirm.

- **MEDIUM**: **`reindex_vectors()` refactoring.** The plan says "groups chunks by file, processes each file group through full path." This is a significant behavioral change — reindex_vectors was previously chunk-order-agnostic, now it's file-grouped. For 229 files that's 229 sequential TEI calls vs. currently ~1 batch call. For 13,500 files this could be significant. The plan should address whether e_meta calls can be batched.

- **MEDIUM**: **`_enrich_noop` approach for handler registry.** Setting `enrich=_enrich_noop` in `content_handlers.py` registry means ALL handlers globally skip enrichment for embeddings. If FTS5 indexing uses the same handler enrich call, it also loses title/tags prefix. The plan states FTS5 still needs enrichment, but the mechanism for the FTS5 path to receive enriched text (vs. the embedding path receiving nooped text) is unspecified.

- **LOW**: **`_embed_meta_component(file_info)` makes 1 TEI call per file.** During bulk `run()` (initial index), this means N sequential TEI calls for N files. For 13,500 files that's 13,500 extra sequential TEI calls. The plan should note this overhead and whether e_meta calls can be batched across files.

- **LOW**: **Schema version `"2"` is a magic string.** Consider using an enum or constant in a shared location.

### Suggestions

- **Specify the metadata-only detection branching logic explicitly** — show the if/elif/else structure for index_file()'s Phase 2.
- **Specify transaction boundaries** for the fast path with a concrete BEGIN/COMMIT block.
- **Keep `_embed_chunks()` pure** — don't have it write to VecComponentStore. Return the e_text vectors and let the caller store them.
- **Clarify the FTS5 enrichment question** — if FTS5 never used enriched text, remove the deprecation comment and delete the enrich function from the embedding path cleanly.
- **Batch e_meta calls** for bulk `run()` — accumulate file_infos, batch their title+tags into a single `encode_batch()` call, then store individually.
- **Add e_text BLOB missing fallback** with explicit behavior: log warning + fall through to full embed for that file.

### Risk Assessment: **HIGH**

This plan touches the most critical hot path in the codebase (indexing pipeline), changes the semantics of `text_hash` (which affects caching correctness), introduces a new storage layer that must be kept in sync with vec0, and has unspecified transaction boundaries. The metadata-only fast path — the core value of the phase — has unspecified detection logic. A bug here means either: (a) metadata changes silently skip re-embedding (search quality regression), or (b) every change triggers full re-embedding (no improvement over current state). Integration testing is critical.

## Plan 03 — Search Logging and Tests

### Summary

Plan 03 adds observability (search_log) and a comprehensive test suite. The test design is well-structured — 23 tests covering the three new primitives plus the critical integration invariant (1 TEI call on metadata-only change). The non-fatal logging approach is correct. However, the plan has a coverage gap around the Plan 02 integration points (fast path detection, transaction boundaries, wipe behavior).

### Strengths

- 23 tests is a good number for this scope — covers unit (VecComponentStore, fusion math, checksums) and integration (metadata-only TEI call count).
- `test_metadata_only_reindex` directly tests the phase's core invariant — exactly 1 `encode_batch` call on metadata-only change.
- Non-fatal search_log with try/except wrapping in `DotMDService.search()` ensures logging never breaks search.
- 10,000 row cap with oldest-first DELETE is simple and bounded.
- All tests runnable without live TEI (mock `encode_batch`) — CI-friendly.

### Concerns

- **HIGH**: **No tests for Plan 02's pipeline integration logic.** The test suite covers Plan 01's primitives and the end-to-end metadata-only path, but skips: (1) fast path detection — does the pipeline correctly distinguish metadata-only vs. body changes vs. no changes? (2) schema wipe — does `_check_schema_version()` correctly clear all state? (3) e_text BLOB missing fallback. (4) transaction atomicity — if fusion fails, are vec0 and VecComponentStore rolled back? (5) `reindex_vectors()` file grouping. These are the highest-risk parts of Plan 02 with zero test coverage.

- **MEDIUM**: `test_metadata_only_reindex` with only 2 tests misses: metadata + body change simultaneously; first deploy after wipe (no e_text BLOBs); weights change → e_fused recomputed with no TEI calls.

- **MEDIUM**: search_log table structure is underspecified — no index on `id`/`timestamp` for the DELETE oldest query. Without indexes, trim requires a full table scan.

- **LOW**: `log_search` should also log which search mode was used (semantic/keyword/graph/hybrid) and whether reranking was applied.

### Suggestions

- **Add 4-5 integration tests for Plan 02's branching logic** in a new `test_pipeline_metadata.py`: metadata-only → fast path (1 TEI call); body change → full re-embed; both change → full re-embed; no change → skip entirely; e_text BLOBs missing → fallback to full embed.
- **Add 1 test for schema wipe behavior** — verify `_check_schema_version()` with stale version clears all state.
- **Add 1 test for weight change** — changing `DOTMD_EMBEDDING_WEIGHTS` recomputes e_fused from stored components without TEI calls.
- **Specify search_log schema fully** including column types, indexes, and what `top_results` contains.
- **Add `mode` and `reranked` columns** to search_log for phases 999.15/999.16.

### Risk Assessment: **MEDIUM**

The observability part (search_log) is low-risk and well-designed. The risk comes from test coverage gaps — the most complex and fragile parts of the phase (pipeline branching, transaction boundaries, wipe behavior) have no automated tests. If Plan 02 has bugs, Plan 03's tests won't catch most of them.

## Consensus Summary

Both Codex and OpenCode reviewed all three plans for Phase 999.12. Both independently converged on HIGH risk for Plan 02 and identified the same root causes.

### Agreed Strengths

- Architecture is sound: the dual-encoder separation correctly addresses the tag-update O(N) → O(1) TEI regression. Both reviewers validated this.
- Wipe-on-deploy via schema_version sentinel is the right migration strategy for this hardware.
- Pure stdlib fusion math (no numpy) is the right call for the AVX2 constraint.
- The three-layer cache ordering (vec_meta text_hash → global embedding_cache → VecComponentStore) is well-designed.
- Plan 01 primitives are clean, isolated, and low-risk.
- Non-fatal search_log design (try/except in DotMDService.search) is correct.
- Test targeting the core invariant (1 encode_batch call on metadata-only change) is the right test to write.

### Agreed Concerns

1. **[HIGH] Metadata-only fast path detection logic is unspecified.** Both reviewers flagged that Plan 02 describes the *what* but not the *how* — the branching logic in `index_file()` that distinguishes body-changed vs. metadata-only vs. no-change is not shown. This is the highest-risk gap.

2. **[HIGH] Transaction boundaries for the fast path are unspecified.** Both reviewers identified the risk of inconsistent state if a TEI call succeeds but a subsequent vec0 write fails. The fast path mutates VecComponentStore, vec0, meta_tracker, FTS5, and graph — no explicit transaction boundary is specified.

3. **[HIGH] `_embed_chunks()` side-effect on VecComponentStore breaks its pure-compute contract.** Both reviewers noted this independently. Codex framed it as a validation risk (chunk_id collision); OpenCode framed it as a contract violation (pure function now writes to storage). The recommended fix is the same: keep `_embed_chunks()` pure, let the caller store e_text.

4. **[HIGH] Test coverage gaps for the highest-risk pipeline logic.** Both reviewers noted Plan 03 covers Plan 01 primitives well but leaves Plan 02's branching logic, schema wipe, e_text BLOB fallback, and transaction atomicity untested.

5. **[MEDIUM] FTS5 enrichment path unverified after noop change.** Both reviewers flagged that the plan states `enrich_with_title_and_tags` is "still needed for FTS5 path" but the actual FTS5 code path is not verified. If FTS5 never used enriched text, the deprecation comment is misleading. If it does, nooping the handler silently breaks FTS5 keyword quality.

6. **[MEDIUM] Bulk run() e_meta call overhead.** Both reviewers independently computed this: 1 extra sequential TEI call per file during bulk indexing. For 13,500 files at ~200ms TEI latency that's ~45 minutes of added overhead on an already multi-day process. Batching e_meta calls across files in bulk mode should be considered.

7. **[MEDIUM] Weight-change recompute path not specified in Plan 02.** D-09 says changing weights is cheap (local math, no TEI), but neither where weight-change is detected nor how stale e_fused vectors are invalidated is specified in any plan.

### Divergent Views

- **Codex** raised the risk of chunk_id collision in the VecComponentStore lookup (e_text BLOB could survive across body changes if chunk_id is reused). **OpenCode** framed the same issue as a function contract violation. Both point to the same fix.
- **Codex** focused more on the completeness of the wipe-on-deploy (ensuring all 7 state components are cleared). **OpenCode** focused more on the FTS5 enrichment verification and bulk indexing overhead quantification.
- **OpenCode** added a cross-cutting concern about search quality rollback: no plan exists to detect if α=0.7/0.3 produces worse retrieval than the current prefix approach. A quality checkpoint after first full rebuild is recommended.
