## Хранение e_fused

| Option | Description | Selected |
|--------|-------------|----------|
| A: e_fused и e_text оба в vec0 | ANN сохранён, но чтение e_text обратно из vec0 — undocumented поведение | |
| B: fusion at query time | Простые writes, но ~80–100 мс брутфорс на запрос без AVX2 | |
| C: e_fused в vec0, компоненты как BLOB | ANN сохранён, компоненты читаются стандартным SQL | ✓ |
| pgvector | Native UPDATE, уже есть voiceprint-postgres на хосте | Деferred → 999.14 |

**User's choice:** C — sqlite-vec с BLOB компонентами.

**Notes:** Пользователь поднял вопрос о pgvector и обнаружил что `voiceprint-postgres` (pgvector/pgvector:pg18) уже работает на хосте. Обсудили coupling с voiceprint-сервисом и необходимость предварительно вынести Postgres в shared service. Решили: остаёмся на sqlite-vec, переезд в backlog (999.14). Ключевой аргумент за sqlite-vec: атомарные транзакции с FTS5/metadata в одном index.db.

## N-векторная схема

| Option | Description | Selected |
|--------|-------------|----------|
| Flat: e_text + e_meta как отдельные таблицы | Проще в реализации, но расширение = migration | |
| N-vector: vec_components(entity_id, component, embedding) | Будущие компоненты добавляются как новые строки, без schema migration | ✓ |

**User's choice:** N-векторная схема с `vec_components` таблицей.

**Notes:** Пользователь предложил эту идею сам, обосновав что метаданные со временем устоятся и появятся новые компоненты для эмбеддинга. Также обсудили геометрическую интерпретацию весов как "аттракторов" — каждый компонент тянет e_fused к своей семантической зоне пространства. Пользователь описал это точно.

## Миграция text_hash

| Option | Description | Selected |
|--------|-------------|----------|
| A: Wipe + rebuild | Чистый slate, несколько дней TEI | ✓ |
| B: cache_version sentinel | Ноль downtime, постепенная деградация/восстановление | |
| C: Переименовать таблицу | Структурный разрыв без rebuild | |
| D: Two-phase reuse | Портировать старые векторы под новые хэши — семантически неверно для этого случая | Отклонён |

**User's choice:** A — wipe on deploy, trickle строит с нуля.

**Notes:** Пользователь уточнил масштаб: не "несколько часов" а **несколько дней** на текущем железе. Option D отклонён по семантическим причинам: старые векторы = embed(title+tags+chunk_text), новые e_text должны быть embed(chunk_text only) — нельзя переиспользовать, иначе метаданные посчитаются дважды при fusion.

## Параметр весов (бывший α)

| Option | Description | Selected |
|--------|-------------|----------|
| A: Хардкод 0.5 | Нет config surface | |
| B: DOTMD_EMBEDDING_WEIGHTS env var | Следует конвенции, тюнинг без code change | ✓ |
| C: Калибровка на eval set | Data-driven, но требует ручной разметки | Отклонён пользователем |
| D: Per-kind веса | После eval-подтверждения | Deferred |
| Авто через кросс-энкодер | Sweep по журналу запросов + reranker как оракул | Deferred → 999.15 |
| Авто через межмодульное согласие | Unsupervised сигнал из semantic+FTS5+graph agreement | Deferred → 999.16 |

**User's choice:** B — env var, дефолт text=0.7,meta=0.3. Автокалибровка в backlog.

**Notes:** Пользователь отклонил eval set подход ("не хочу возиться с разметкой"). Предложил автоматическую калибровку через accumulated query log — идея кросс-энкодера и межмодульного согласия понравилась. Решено: добавить query logging в эту фазу как фундамент, автокалибровку отложить на 999.15 и 999.16 когда накопится журнал.
