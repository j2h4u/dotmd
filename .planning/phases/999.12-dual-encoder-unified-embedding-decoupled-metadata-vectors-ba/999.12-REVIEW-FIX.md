## Fixes Applied

### CR-01: WIPE_IN_PROGRESS sentinel + crash-safe wipe protocol
**Status:** FIXED — commit `d2df460`

Implemented two-sentinel crash-safe wipe in `_check_schema_version()`:
- Step 0 now writes `sentinel = 'WIPE_IN_PROGRESS'` before any wipe step
- Steps 1–6 wipe idempotently (each commits internally, as before)
- Step 7 writes `sentinel = SCHEMA_VERSION` (replaces WIPE_IN_PROGRESS)
- On next startup: `WIPE_IN_PROGRESS` is detected and all wipe steps are re-run (idempotent)

Docstring updated to describe the actual crash-safe semantics instead of the
previously false claim of atomic `BEGIN…COMMIT`.

### WR-01: `count()` guard for pre-999.12 databases with partial vec_components data
**Status:** FIXED — commit `d2df460`

Added `vec_components.count()` check inside the `stored_version is None` branch:
- `count() == 0` → fresh DB: write sentinel and return (no wipe)
- `count() > 0` → orphaned partial 999.12 deployment data: fall through to wipe path

This prevents `_embed_chunks()` Layer-1 cache from returning stale vectors from a
partial previous deployment that was interrupted before writing the sentinel.

### WR-03: Explicit `conn.commit()` before `add_chunks` in `_save_and_embed_chunks`
**Status:** FIXED — commit `d43902c`

Added `self._conn.commit()` after the per-file `VecComponentStore.store()` loop and
before `_vector_store.add_chunks()`. With `isolation_level=None` (autocommit) each
INSERT auto-commits, but `VecComponentStore.store()` docstring states "caller owns
the commit boundary". The explicit flush makes the intent unambiguous and prevents
any future regression if the connection mode changes.

---

### WR-04: `_check_weights_changed` — single `delete_all()` before loop, `overwrite=False` per file
**Status:** FIXED — commit `c6457dc`

- Added `self._vector_store.delete_all()` once before the batch loop
- Changed `overwrite=True` → `overwrite=False` on the per-file `add_chunks()` call

Previously, `overwrite=True` triggered `DELETE FROM vec_chunks` + `DELETE FROM vec_meta`
on every file iteration, which was both semantically wrong (table empty between iterations)
and extremely slow on large indexes (full virtual table scan per file). Now vec0 is cleared
once then filled per-file, which is correct and efficient.

### WR-05: `_build_file_meta_from_fileinfo` — `None` guard on `frontmatter`
**Status:** FIXED — commit `88ca3bf`

Added `fm = fi.frontmatter or {}` at the start of the loop body, matching the same
guard pattern already used in `_embed_meta_component` and `reindex_vectors`. Prevents
`AttributeError` on `FileInfo` objects where `frontmatter` is `None` rather than `{}`.
