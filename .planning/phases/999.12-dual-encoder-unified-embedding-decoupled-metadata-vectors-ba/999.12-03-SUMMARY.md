# Phase 999.12 Plan 03: Search Logging and Tests Summary

Search logging infrastructure (search_log table + log_search() method) added to the indexing pipeline and wired into DotMDService.search(). Comprehensive test coverage written for Plan 01 and Plan 02 primitives. Five pre-existing bugs discovered and fixed during test execution.

## What Was Built

### Task 1: search_log table + log_search() method
- `CREATE TABLE IF NOT EXISTS search_log` with `mode TEXT NOT NULL DEFAULT 'hybrid'` and `reranked INTEGER NOT NULL DEFAULT 0` columns for Phase 999.15/999.16 calibration use
- `CREATE INDEX search_log_id_idx ON search_log(id)` for efficient DELETE...ORDER BY id trim
- `log_search()` method: writes one row per search, trims at 10,000 row cap, logs parsed weights (not raw env var string), wrapped in try/except (non-fatal)
- Constant `_SEARCH_LOG_MAX_ROWS = 10_000` limits table to ~3 MB at ~300 bytes/row

### Task 2: DotMDService.search() logging wire-up
- `self._pipeline.log_search()` called in `_execute_search()` after `build_search_results`
- Passes `original_query`, `parsed_embedding_weights`, top results as `[{chunk_id, score, engine}]` dicts
- `mode` normalized to str via `str(mode)`, `reranked` derived from `bool(self._reranker)`
- Outer try/except ensures search() never fails due to logging errors

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `_embed_existing_chunks` BEGIN/COMMIT transaction errors**
- **Found during:** Task 6 (test_pipeline_metadata.py test execution)
- **Issue:** Both branches of `_embed_existing_chunks` wrapped operations in `BEGIN...COMMIT`, but `add_chunks()` calls `conn.commit()` internally, consuming the transaction. Final `COMMIT` then failed with "cannot commit - no transaction is active".
- **Fix:** Removed outer BEGIN/COMMIT in both branches. Added explicit `conn.commit()` before `add_chunks()` to flush VecComponentStore stores, letting `add_chunks()` own its own commit.
- **Files modified:** `backend/src/dotmd/ingestion/pipeline.py`
- **Commit:** be0cb48

**2. [Rule 1 - Bug] `_check_schema_version` BEGIN/COMMIT transaction error**
- **Found during:** Task 6 (test_schema_version_wipe_clears_all_state)
- **Issue:** Same pattern — schema wipe wrapped in BEGIN/COMMIT but each sub-operation (embedding_cache.clear, vec_components.delete_all, vector_store.delete_all, chunk_tracker.clear, meta_tracker.clear) calls conn.commit() internally.
- **Fix:** Removed outer BEGIN/COMMIT. Each operation commits individually; schema_version sentinel written last for crash-safety (re-wipe on next startup if crash before sentinel write).
- **Files modified:** `backend/src/dotmd/ingestion/pipeline.py`
- **Commit:** be0cb48

**3. [Rule 1 - Bug] `sqlite_vec.add_chunks` lastrowid unreliable for INSERT OR IGNORE in autocommit mode**
- **Found during:** Task 8 (test_m2m_shared_chunk_behavior_documented)
- **Issue:** `cur.lastrowid` returns the rowid of the PREVIOUS successful INSERT (not 0) when INSERT OR IGNORE is a no-op in `isolation_level=None` autocommit mode. This caused the code to attempt `INSERT INTO vec0 (rowid=N)` for a chunk that already existed — UNIQUE constraint violation.
- **Fix:** Changed `if cur.lastrowid:` to `if cur.rowcount and cur.lastrowid:`. `rowcount == 0` correctly detects no-op regardless of connection mode.
- **Files modified:** `backend/src/dotmd/storage/sqlite_vec.py`
- **Commit:** 011aa8a

**4. [Rule 1 - Bug] `_embed_existing_chunks(model_switch=True)` uses stale VecComponentStore e_text (HIGH-2)**
- **Found during:** Task 8 (test_embed_existing_chunks_model_switch_does_not_use_cached_etext)
- **Issue:** `model_switch=True` called `_embed_chunks()` which reads VecComponentStore as Layer 1 cache. During model switch, VecComponentStore contains e_text from the OLD model. The cache hit bypassed TEI entirely, defeating the purpose of model_switch=True.
- **Fix:** For `model_switch=True`, bypass `_embed_chunks()` and call `self._semantic_engine.encode_batch(texts)` directly. Computes text_hashes separately via blake3.
- **Files modified:** `backend/src/dotmd/ingestion/pipeline.py`
- **Commit:** 011aa8a

**5. [Test adaptation] `embedding_cache` also needed clearing in fallback test**
- **Found during:** Task 6 (test_metadata_only_with_missing_etext_falls_back_to_full_embed)
- **Issue:** Test wiped VecComponentStore but left embedding_cache intact. The global cache (Layer 2) hit on the chunk text and bypassed TEI — test got 1 encode_batch call (just e_meta), not ≥2.
- **Fix:** Added `pipeline._embedding_cache.clear()` alongside `pipeline._vec_components.delete_all()` to force true fallback-to-TEI path.
- **Files modified:** `backend/tests/ingestion/test_pipeline_metadata.py`
- **Commit:** be0cb48

**6. [Test adaptation] `pipeline.run()` → `pipeline.index(settings.data_dir)`**
- **Found during:** All integration tests
- **Issue:** Plan specified `pipeline.run()` but the actual API is `pipeline.index(directory)`.
- **Fix:** All test helpers use `pipeline.index(settings.data_dir)` throughout.
- **No additional commit** (caught before committing)
