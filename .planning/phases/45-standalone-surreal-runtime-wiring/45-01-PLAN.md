# Phase 45 Plan 01: Standalone SurrealDB Runtime Wiring and Smoke

## Objective

Make standalone SurrealDB a real runtime search backend, not only a shadow-run
override, then prove the user-facing surfaces can run against it before retrying
production cutover.

This phase does not delete the old stack. It creates the bounded bridge needed
after Phase 44 NO-GO: runtime search can use standalone SurrealDB retrieval
engines while metadata/read/trickle boundaries remain explicit.

## Decisions

- Runtime selection is config-gated by `search_backend`.
- `search_backend=sqlite` remains the default until cutover is explicitly
  accepted.
- `search_backend=surreal` means SurrealDB-backed retrieval, not full storage
  replacement. Metadata hydration, read paths, and trickle writes remain old
  stack responsibilities until this phase records a separate decision.
- No hidden Falkor graph-direct fallback is allowed in Surreal retrieval mode.

## Tasks

### Task 1: Service Runtime Wiring

Add settings and startup wiring so `DotMDService` can replace semantic, keyword,
and graph-direct retrieval engines with standalone SurrealDB engines.

Acceptance:

- startup fails fast when required Surreal retrieval config is incomplete;
- auth config is validated and passed through `SurrealConnection`;
- Surreal retrieval mode disables seed-based legacy graph enrichment;
- default runtime remains unchanged.

### Task 2: Controlled Runtime Smoke

Run bounded smoke against the populated standalone database
`dotmd/phase43_refresh_20260618g` without cutting over production traffic.

Acceptance:

- service init succeeds with `search_backend=surreal`;
- at least one search returns valid source refs and scores;
- failures include exact failing surface and error text;
- smoke duration and command are recorded.

### Task 3: Surface Matrix and Trickle Decision

Decide what can be smoked now through MCP/API/CLI and what remains blocked by
write-path semantics.

Acceptance:

- MCP/API/CLI smoke either passes or has a concrete blocker;
- trickle is explicitly classified as old-stack-only, Surreal-write-ready, or
  cutover-blocking;
- Phase 44 retry criteria are updated from evidence, not assumptions.

## Verification

- `uv run ruff check` on changed files.
- Focused service/config/storage tests.
- Full non-live backend test suite with `tests/e2e` excluded.
- Live e2e only after a reachable MCP endpoint is intentionally started or the
  production container is restarted as a planned deployment action.
