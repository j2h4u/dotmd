# Phase 47 Plan 01: Legacy Stack Removal

## Objective

Remove SQLite/sqlite-vec/FTS5/FalkorDB/LadybugDB as production runtime
backends after the accepted standalone SurrealDB cutover. Old-stack artifacts
may remain only where they are explicitly migration evidence or temporary
internal cache/change-detection scaffolding scheduled for deletion in this
phase.

## Constraints

- Do not add compatibility fallbacks.
- Do not preserve `search_backend=sqlite` as an operator choice.
- Do not run destructive production indexing commands while the container is
  running.
- Keep rollback snapshots until the first Phase 47 production smoke passes.
- Any operation expected to run longer than 120 seconds must log progress or be
  run with an external timeout/monitor.

## Tasks

1. Runtime switch removal
   - Remove `search_backend` as a production backend selector.
   - Make SurrealDB retrieval settings mandatory for runtime startup.
   - Remove `falkordb_url` runtime validation and production graph backend
     identity from current status output.
   - Verify focused config tests.

2. CLI/admin surface cleanup
   - Rewrite `dotmd status` so it reports SurrealDB runtime/counts instead of
     local SQLite/Falkor internals.
   - Remove or fail-fast old-stack admin commands such as `reindex fts5`,
     `reindex graph`, `reset model`, and `reset strategy` unless they are
     replaced with Surreal-native equivalents in the same commit.
   - Verify CLI smoke against the running container or focused CLI tests.

3. Service and pipeline collapse
   - Initialize Surreal retrieval engines unconditionally.
   - Remove branches that instantiate old local retrieval engines for
     production search.
   - Keep any remaining SQLite use clearly internal and temporary until the
     metadata/change-detection layer is replaced or deleted.
   - Verify service search and trickle focused tests.

4. Test suite cleanup
   - Delete tests that only enforce old-stack backend choices.
   - Rewrite tests that should now assert Surreal-only behavior.
   - Keep migration/devtools tests only when they protect rollback evidence or
     one-off migration tooling still retained in repo.

5. Module and dependency deletion
   - Once imports are gone, delete dead old-stack modules and dependencies:
     sqlite-vec runtime store, FTS5 search engine, FalkorDB graph store, and
     LadybugDB docs/config references.
   - Remove deployment/env assumptions for FalkorDB and old local DB files.

6. Documentation and data cleanup
   - Update README, architecture docs, MCP docs, env examples, and AGENTS.md to
     present standalone SurrealDB as the only production storage backend.
   - After final production smoke, delete rollback/snapshot data that is no
     longer needed.

## Verification

- `UV_LINK_MODE=hardlink uv run ruff check ...` over changed Python files.
- Focused pytest for config/service/pipeline/CLI changes.
- Live smoke:
  - `docker exec dotmd curl -fsS http://127.0.0.1:8080/health`
  - no-rerank Surreal search smoke
  - controlled trickle insert/edit/delete smoke if pipeline write logic changes

## Success Criteria

- Production runtime has no operator-selectable SQLite/Falkor fallback.
- Public CLI/status/docs no longer describe SQLite/sqlite-vec/FTS5/FalkorDB as
  active production storage.
- Old-stack code that remains is explicitly temporary migration/cache scaffolding
  with a tracked deletion path inside Phase 47.
- Tests pass for the edited surface.
