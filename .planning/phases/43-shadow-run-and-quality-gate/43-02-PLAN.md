## Plan-Level Decisions

**D43-02-A â€” Baseline capture binds the old stack to the rehearsal path by constructing a fresh `DotMDService` from a path-overridden `Settings` clone; it never mutates live config (addresses consensus HIGH "baseline capture mechanics underspecified").**
The runner builds the baseline service exactly like `surreal_migration_runner.py` builds its config: explicit, local, single-purpose. Concrete mechanism:
- Load base settings via `load_settings()` (43-02 already imports the service layer), then produce an isolated clone with `Settings.model_copy(update={"index_dir": Path(baseline_rehearsal_path)})`. `Settings.index_db_path` derives from `index_dir` as `index_dir / "index.db"`, so overriding `index_dir` points the entire old stack (SQLite metadata + FTS5 + sqlite-vec) at the copied snapshot without touching env vars, the live TOML, or the production volume.
- Pass the cloned settings into a fresh `DotMDService(settings=cloned_settings)` constructed once per capture process (never per query, per AGENTS.md "never reload indexes per request"). Read-only search calls (`_collect_candidate_pool` with no engine overrides) produce baseline `EvalResult` rows.
- The runner does NOT call `Settings.validate_for_runtime()` on the clone (that method requires the production `/dotmd-index` and `/mnt` constants and would reject a rehearsal path); plain `DotMDService` construction does not invoke runtime validation, so the rehearsal index loads cleanly.
- No write path is invoked: no `dotmd index`, no trickle start, no `index --force`. Baseline capture is search-only against an already-built copied index.

**D43-02-B â€” Rehearsal-path isolation is enforced, not assumed (addresses OpenCode HIGH isolation requirement).**
Before constructing the baseline service, the runner enforces, fail-closed:
1. `--baseline-rehearsal-path` is required for `--capture-baseline`; absence is a hard error.
2. The resolved rehearsal path MUST NOT equal (or be a parent/child of) the production `DOTMD_INDEX_DIR`. The runner reads the production index dir from the environment (`DOTMD_INDEX_DIR`) and from the `RUNTIME_INDEX_DIR` constant (`/dotmd-index`) in `dotmd.core.config`, resolves both with `Path.resolve()`, and refuses to run if the rehearsal path resolves to, contains, or is contained by either. This prevents accidentally pointing baseline capture at the live volume.
3. The rehearsal `index.db` MUST pass `PRAGMA integrity_check` (opened read-only via `sqlite3.connect("file:...?mode=ro", uri=True)`) before any search call. A failing or `ok`-absent integrity result is a hard error. This proves the copied snapshot is a real, intact SQLite database rather than a truncated/symlinked stand-in.

**D43-02-C â€” Rehearsal-path contract: a valid copy is a real intact SQLite snapshot, not a symlink (addresses cross-cutting "Rehearsal Path Contract" gap).**
A valid rehearsal path is a directory containing an `index.db` that (a) is a regular file, not a symlink (`Path.is_symlink()` is False), (b) is not the production index dir per D43-02-B, and (c) passes `PRAGMA integrity_check`. The runbook documents that operators produce it with the SQLite backup API or a filesystem copy (`cp -r` of a stopped-container volume snapshot), consistent with the Phase 38 copied-snapshot evidence policy. The runner validates the contract; it does not produce the copy. To make a stale/truncated copy visible (addresses cycle-2 MEDIUM "WAL/snapshot completeness under-validated"), the runner records the rehearsal `index.db` byte size and mtime in the source-capture manifest alongside the integrity result; a full FalkorDB+WAL dump/restore is out of scope because graph evidence is excluded (D43-02-G).

**D43-02-D â€” Flag surface is reduced by folding target-derived tuning knobs into a single `--candidate-config-json` (resolves the divergent "split vs reduce" view in favor of reduction; addresses "27-flag scope creep").**
`--embedding-dimension`, `--hnsw-ef`, `--top-k`, and `--pool-size` are removed as standalone flags. They are read from a `--candidate-config-json` descriptor (a small JSON file the operator derives from the prepared Surreal target), because they are properties of the candidate target, not runner behavior. The runner keeps the connection flags (`--target-url`, `--target-namespace`, `--target-database`) and the orchestration/artifact flags. This keeps 43-02 a single plan (no split) while shrinking the CLI surface and tying retrieval tuning to the target. The exact field schema and which fields route to engine construction vs the capture loop is pinned by D43-02-H: `build_surreal_native_engine_overrides(connection, settings, embedding_dimension=..., hnsw_ef=...)` receives ONLY `embedding_dimension` and `hnsw_ef` from the parsed candidate-config descriptor (the only two params that function accepts); `top_k` and `pool_size` are capture-search-loop inputs and are NOT passed to `build_surreal_native_engine_overrides`.

**D43-02-E â€” `--metrics-replay-queries` is a `type=Path` JSONL descriptor (addresses "type ambiguity" concern).**
`--metrics-replay-queries` is pinned to `type=Path` pointing at a JSONL file whose rows describe the larger production-derived replay window used for latency/build/store/memory evidence (distinct from the 16-query golden quality corpus). The runner loads it with the same strict line-numbered JSON/JSONL validation used for other artifacts and fails closed on malformed rows or a missing file when metric capture is requested.

**D43-02-F â€” Sentinel stripping is proven by named test cases, never reaching the Phase 40 loader (addresses Codex HIGH on sentinel/Phase 40-loader interaction).**
`load_shadow_acceptance_ledger()` strips every `record_type="phase43_ledger_metadata"` row and passes only real `query_id`/`accepted_by`/`accepted_reason` rows into Phase 40 acceptance loading. The Phase 40 loader never sees a sentinel row (which lacks those required fields and would otherwise raise). This is pinned by the named tests in Task 1.

**D43-02-G â€” Baseline capture is semantic+FTS-only; graph engines are disabled for the rehearsal window and graph-category golden queries are excluded from the shadow comparison (addresses cycle-2 consensus HIGH "baseline graph state is not bound to the rehearsal window").**
Source-confirmed root cause: `Settings.model_copy(update={"index_dir": ...})` (D43-02-A) redirects only the SQLite/FTS5/sqlite-vec side. The FalkorDB graph backend is built by `_create_graph_store()` (`backend/src/dotmd/ingestion/pipeline.py:161`) as `FalkorDBGraphStore(url=settings.falkordb_url, graph_name="dotmd")` â€” bound to `settings.falkordb_url` (`backend/src/dotmd/core/config.py:208`) and a hardcoded `graph_name="dotmd"`, independent of `index_dir`. So a baseline `DotMDService` built from the cloned settings would point SQLite at the rehearsal copy while graph engines still reach the LIVE FalkorDB container (breaking snapshot isolation), or crash if the clone lacks a graph URL (breaking completeness). D43-02-A's "entire old stack" claim is therefore incomplete for graph/hybrid queries.

Resolution chosen: **option (b) â€” disable graph engines for baseline capture and scope Phase 43 to semantic+FTS quality evidence.** Option (a) (a dedicated isolated graph name/target) is rejected because `graph_name="dotmd"` is hardcoded in `_create_graph_store()`; isolating it would require editing production startup wiring, which violates the evidence-only / no-cutover scope of this phase. Option (b) is achievable entirely through the existing per-call seam with no production change.

Concrete mechanism (grounded in `backend/src/dotmd/api/service.py:1339` `_collect_candidate_pool`, which selects which engines fire by `SearchMode`):
- Baseline capture in `build_baseline_service` / the baseline search loop restricts the search `mode` so that graph-direct and graph-enrichment engines never fire. `_collect_candidate_pool` runs graph-direct only for `SearchMode.GRAPH`/`SearchMode.HYBRID`; baseline capture therefore drives the old stack with semantic+keyword(FTS) modes only and never with `GRAPH`/`HYBRID`, so `self._graph_direct_engine` / `self._graph_engine` are not invoked and no FalkorDB connection is opened during baseline capture. This is structurally enforced by the captured query set, not by a runtime monitor (per AGENTS feedback "invariant by construction").
- The golden-query corpus consumed for the shadow comparison is filtered to NON-graph categories for BOTH baseline and candidate so the two sides are compared on identical, graph-free query sets. Graph-category golden queries are explicitly EXCLUDED from the shadow comparison and are recorded as excluded (with a count and the excluded query ids) in the source-capture manifest and the shadow summary, so a graph-category regression is visibly out-of-scope rather than silently missed.
- The runner exposes this as a fail-closed contract: `build_baseline_service` / the baseline capture path asserts no graph engine is exercised (a baseline capture that would invoke `GRAPH`/`HYBRID` mode is a hard error), and the source-capture manifest carries an explicit `graph_engines_disabled: true` / `excluded_query_categories: ["graph"]` (or equivalent) field so 43-03's acceptance ledger and golden-query set reflect the exclusion.

**D43-02-H â€” `--candidate-config-json` has an explicit fail-closed field schema separating engine-override params from capture-loop params (addresses cycle-2 MEDIUM "`--candidate-config-json` schema underspecified", source-confirmed).**
Source-confirmed: `build_surreal_native_engine_overrides()` (`backend/src/dotmd/search/surreal_native.py:18`) accepts ONLY `embedding_dimension` (required `int`) and `hnsw_ef` (`int`, default `DEFAULT_HNSW_EF`). `top_k` and `pool_size` are NOT engine-construction params â€” they are inputs to the capture search loop. D43-02-D conflated the two routes. `load_candidate_config(path)` is therefore given an explicit schema:
- `embedding_dimension`: required, positive `int` â†’ fed to `build_surreal_native_engine_overrides(..., embedding_dimension=...)`.
- `hnsw_ef`: optional, positive `int`, defaults to `DEFAULT_HNSW_EF` â†’ fed to `build_surreal_native_engine_overrides(..., hnsw_ef=...)`.
- `top_k`: required, positive `int` â†’ consumed by the capture search loop only (NOT passed to `build_surreal_native_engine_overrides`).
- `pool_size`: required, positive `int` â†’ consumed by the capture search loop only (NOT passed to `build_surreal_native_engine_overrides`).
`load_candidate_config()` validates every present field is a positive integer of the correct type, applies the `hnsw_ef` default when absent, raises a `ValueError` naming the offending field on a non-positive/non-integer value or a missing required field, and REJECTS unknown keys (fail-closed) so malformed target-derived tuning cannot silently run. Pinned by named tests in Task 1.
