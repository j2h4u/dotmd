<objective>
Create the Phase 43 shadow-run runner that ties Phase 40 evaluation, Phase 41 source-capture evidence, Phase 42 explicit Surreal overrides, and Phase 43 metrics into one bounded artifact bundle.

Purpose: The milestone needs old-stack baseline evidence and Surreal candidate evidence on the same production-derived window without adding cutover wiring or a fallback backend. This runner gives executors one reproducible command surface for that evidence.
Output: A tested devtool runner, focused runner tests, and an operator runbook.
</objective>

<decisions>
## Plan-Level Decisions

**D43-02-A — Baseline capture binds the old stack to the rehearsal path with a `Settings.model_copy` index_dir clone; it never mutates live config (addresses consensus HIGH "baseline capture mechanics underspecified"). SUPERSEDED in part by D43-02-I: the clone is consumed by the graph-free engine factory, NOT by a `DotMDService` constructor.**
The runner derives the rehearsal-bound settings exactly like `surreal_migration_runner.py` builds its config: explicit, local, single-purpose. Concrete mechanism:
- Load base settings via `load_settings()` (43-02 already imports the service layer), then produce an isolated clone with `Settings.model_copy(update={"index_dir": Path(baseline_rehearsal_path)})`. `Settings.index_db_path` derives from `index_dir` as `index_dir / "index.db"`, so overriding `index_dir` points the SQLite/FTS5/sqlite-vec side of the old stack at the copied snapshot without touching env vars, the live TOML, or the production volume. `falkordb_url` (`backend/src/dotmd/core/config.py:208`) carries through the clone UNCHANGED — this is exactly why a `DotMDService`/`IndexingPipeline` constructed from the clone would still reach the live FalkorDB container, and is the reason baseline capture must NOT construct them (see D43-02-I).
- The cloned settings are consumed ONLY by `build_baseline_semantic_keyword_engines(rehearsal_settings)` (D43-02-I), which builds the SQLite/FTS5/sqlite-vec engines directly. The runner does NOT construct `DotMDService` or `IndexingPipeline` for baseline capture, because both eagerly build `FalkorDBGraphStore` (verified: `DotMDService.__init__` `backend/src/dotmd/api/service.py:264` → `IndexingPipeline.__init__` `backend/src/dotmd/ingestion/pipeline.py:246` → `_create_graph_store()` `pipeline.py:161` → `FalkorDBGraphStore.__init__` `backend/src/dotmd/storage/falkordb_graph.py:47-63`, which connects AND issues `CREATE INDEX` writes against the live `dotmd` graph at construction time).
- The runner does NOT call `Settings.validate_for_runtime()` on the clone (that method requires the production `/dotmd-index` and `/mnt` constants and would reject a rehearsal path).
- No write path is invoked: no `dotmd index`, no trickle start, no `index --force`. Baseline capture is search-only against an already-built copied index.

**D43-02-I — Baseline pool is built graph-free by construction: a semantic+FTS-only engine factory that NEVER constructs `DotMDService` / `IndexingPipeline` / `FalkorDBGraphStore` (resolves cycle-3 consensus HIGH "baseline service construction crosses the live FalkorDB boundary"; supersedes the `DotMDService`-construction mechanism of D43-02-A/G).**
Source-confirmed boundary problem: constructing `DotMDService(settings=clone)` eagerly builds the FalkorDB graph store at construction time (`service.py:264` → `pipeline.py:246` → `_create_graph_store` `pipeline.py:161` → `FalkorDBGraphStore.__init__` `falkordb_graph.py:47-53` connects, `:60-63` issues `CREATE INDEX FOR (n:{label}) ON (n.id)` for `File/Section/Entity/Tag/Node`). The cloned settings do not override `falkordb_url`, so this targets the LIVE container — before any `SearchMode` is chosen. Restricting `SearchMode` (the old D43-02-G mechanism) cannot help, because the connection + index writes happen in the constructor, not in the search call. This falsified D43-02-G's "no FalkorDB connection opened" truth, threat row T-43-04c, and the `*_has_no_side_effects` test premise.

Chosen mechanism (the cleanest in-phase option):
- `build_baseline_semantic_keyword_engines(rehearsal_settings)` opens the rehearsal `index.db` with a single `sqlite3.connect` (WAL, `sqlite_vec.load`) and constructs ONLY the two graph-free engines the old stack uses for semantic+FTS retrieval, with the exact constructor calls `IndexingPipeline.__init__` makes (`backend/src/dotmd/ingestion/pipeline.py:241,297,304`):
  - `SemanticSearchEngine(SQLiteVecVectorStore(conn, table_name=vec_table), settings.embedding_model, score_floor=settings.semantic_score_floor, embedding_url=settings.embedding_url, tei_batch_size=settings.tei_batch_size, use_prefix=settings.needs_embedding_prefix, query_instruction=settings.query_instruction)`.
  - `FTS5SearchEngine(conn, table_name=fts_table)`.
  - Table names are derived with the same `(chunk_strategy, embedding_model)` suffix logic the pipeline uses (`_model_to_table_suffix` / `chunks_fts_{strategy}` / `vec_chunks_{strategy}{suffix}`), reused from `dotmd.ingestion.pipeline`, so the engines read the rehearsal index's real tables.
- `collect_semantic_keyword_pool(semantic_engine, keyword_engine, search_query, pool_size, fusion_k)` runs the two engines and fuses with `fuse_results({"semantic": ..., "keyword": ...}, k=fusion_k)` — the same RRF call `_collect_candidate_pool` uses (`service.py:1407`), with NO `graph_direct`/`graph` peer. This is a real graph-free path built from existing public symbols; it does not depend on any combined `SearchMode` enum value (the enum has only `SEMANTIC`/`KEYWORD`/`GRAPH`/`HYBRID`, verified `backend/src/dotmd/core/models.py:20-26`).
- Because `FalkorDBGraphStore` is never instantiated and `DotMDService`/`IndexingPipeline` are never constructed for baseline capture, the runner opens NO FalkorDB connection and issues NO `CREATE INDEX` during baseline pool construction — the isolation invariant now holds by construction, satisfying the phase read-only/no-live-mutation scope (T-43-09).

Rejected alternatives:
- *Inject a null/no-op graph store into the pipeline/service.* The only injection seam is `_create_graph_store()` (`pipeline.py:161`), production startup wiring. Editing it (or monkeypatching it in a devtool) to swap a `NullGraphStore` would change/forge production construction behavior and is out of the evidence-only / no-cutover scope; it also still constructs the full `IndexingPipeline` (trickle indexer, source lifecycle, extractors) the baseline pool does not need.
- *Restrict `SearchMode` on a `DotMDService` built from the clone.* Rejected — the FalkorDB connect + `CREATE INDEX` write happen in the constructor, before any `SearchMode` gate, so this does not isolate the boundary at all (the cycle-2 fix that this decision corrects).

**D43-02-J — Rehearsal index identity is compared against the operator-supplied expected manifest and FAILS CLOSED before baseline capture (addresses cycle-3 MEDIUM "rehearsal WAL/snapshot identity only made visible, not fail-closed").**
Recording rehearsal `index.db` size + mtime (D43-02-C) makes a stale copy visible but does not prove it matches the expected import identity. `assert_rehearsal_identity_matches_manifest(rehearsal_settings, expected_manifest)` therefore queries the rehearsal `index.db` read-only and compares, fail-closed BEFORE any baseline search call: (a) imported chunk count for `settings.chunk_strategy`, (b) embedding/vector row count for `(chunk_strategy, embedding_model)`, (c) chunk strategy, and (d) embedding model, each against the corresponding field in the operator-supplied `source-capture-expected.json` (D43-02-K). Any mismatch raises a `ValueError` naming the offending field and aborts the run before baseline capture begins; no partial baseline evidence is written. Pinned by `test_rehearsal_identity_mismatch_fails_closed` and `test_rehearsal_identity_match_passes`.

**D43-02-K — The operator-supplied EXPECTED identity descriptor is a distinct immutable input file, separate from the runner-produced evidence (addresses cycle-3 MEDIUM "`source-capture.json` is both immutable preflight input AND runner-produced output").**
The expected-identity descriptor the operator derives from the Phase 41 import evidence is `source-capture-expected.json` (loaded by `load_expected_source_manifest(path)` into an `ExpectedSourceManifest` carrying `chunk_strategy`, `embedding_model`, `import_id`, `expected_chunk_count`, `expected_embedding_count`). It is consumed read-only by `preflight_candidate_target` (D43-03-D) and `assert_rehearsal_identity_matches_manifest` (D43-02-J) and is NEVER rewritten by the runner. The runner-produced evidence file `source-capture.json` is generated/echoed by `run_shadow_run`: it copies the expected fields, adds the rehearsal `index.db` size/mtime/integrity result, the graph-exclusion marker (D43-02-G), and the actual observed counts. `--source-capture-manifest-json` (the input flag) points at `source-capture-expected.json`; the produced `source-capture.json` is an output artifact. Validating against an input the runner never rewrites closes the "validate against an artifact it later rewrites" gap. Pinned by `test_expected_manifest_input_is_never_rewritten`.

**D43-02-B — Rehearsal-path isolation is enforced, not assumed (addresses OpenCode HIGH isolation requirement).**
Before constructing the baseline service, the runner enforces, fail-closed:
1. `--baseline-rehearsal-path` is required for `--capture-baseline`; absence is a hard error.
2. The resolved rehearsal path MUST NOT equal (or be a parent/child of) the production `DOTMD_INDEX_DIR`. The runner reads the production index dir from the environment (`DOTMD_INDEX_DIR`) and from the `RUNTIME_INDEX_DIR` constant (`/dotmd-index`) in `dotmd.core.config`, resolves both with `Path.resolve()`, and refuses to run if the rehearsal path resolves to, contains, or is contained by either. This prevents accidentally pointing baseline capture at the live volume.
3. The rehearsal `index.db` MUST pass `PRAGMA integrity_check` (opened read-only via `sqlite3.connect("file:...?mode=ro", uri=True)`) before any search call. A failing or `ok`-absent integrity result is a hard error. This proves the copied snapshot is a real, intact SQLite database rather than a truncated/symlinked stand-in.

**D43-02-C — Rehearsal-path contract: a valid copy is a real intact SQLite snapshot, not a symlink (addresses cross-cutting "Rehearsal Path Contract" gap).**
A valid rehearsal path is a directory containing an `index.db` that (a) is a regular file, not a symlink (`Path.is_symlink()` is False), (b) is not the production index dir per D43-02-B, and (c) passes `PRAGMA integrity_check`. The runbook documents that operators produce it with the SQLite backup API or a filesystem copy (`cp -r` of a stopped-container volume snapshot), consistent with the Phase 38 copied-snapshot evidence policy. The runner validates the contract; it does not produce the copy. The runner records the rehearsal `index.db` byte size and mtime in the produced `source-capture.json` alongside the integrity result for auditability, AND (per the cycle-3 fail-closed requirement, D43-02-J) compares the rehearsal index's chunk count, embedding count, chunk strategy, and embedding model against the operator-supplied `source-capture-expected.json` and aborts before baseline capture on any mismatch — so a stale/truncated/wrong copy is not merely visible but rejected. A full FalkorDB+WAL dump/restore is out of scope because graph evidence is excluded (D43-02-G/I).

**D43-02-D — Flag surface is reduced by folding target-derived tuning knobs into a single `--candidate-config-json` (resolves the divergent "split vs reduce" view in favor of reduction; addresses "27-flag scope creep").**
`--embedding-dimension`, `--hnsw-ef`, `--top-k`, and `--pool-size` are removed as standalone flags. They are read from a `--candidate-config-json` descriptor (a small JSON file the operator derives from the prepared Surreal target), because they are properties of the candidate target, not runner behavior. The runner keeps the connection flags (`--target-url`, `--target-namespace`, `--target-database`) and the orchestration/artifact flags. This keeps 43-02 a single plan (no split) while shrinking the CLI surface and tying retrieval tuning to the target. The exact field schema and which fields route to engine construction vs the capture loop is pinned by D43-02-H: `build_surreal_native_engine_overrides(connection, settings, embedding_dimension=..., hnsw_ef=...)` receives ONLY `embedding_dimension` and `hnsw_ef` from the parsed candidate-config descriptor (the only two params that function accepts); `top_k` and `pool_size` are capture-search-loop inputs and are NOT passed to `build_surreal_native_engine_overrides`.

**D43-02-E — `--metrics-replay-queries` is a `type=Path` JSONL descriptor with a defined per-row schema (addresses "type ambiguity" concern + cycle-3 LOW "`--metrics-replay-queries` per-row JSONL schema undefined").**
`--metrics-replay-queries` is pinned to `type=Path` pointing at a JSONL file whose rows describe the larger production-derived replay window used for latency/build/store/memory evidence (distinct from the 16-query golden quality corpus). Each row is a JSON object with the REQUIRED fields `query_id` (non-empty string, unique within the file) and `query` (non-empty string); any other fields are ignored. The runner loads it with strict line-numbered JSON/JSONL validation and fails closed (with the offending line number) on malformed rows, a row missing `query_id` or `query`, a duplicate `query_id`, or a missing file when metric capture is requested. The runbook documents this minimal row schema (D43-02-E).

**D43-02-F — Sentinel stripping is proven by named test cases, never reaching the Phase 40 loader (addresses Codex HIGH on sentinel/Phase 40-loader interaction).**
`load_shadow_acceptance_ledger()` strips every `record_type="phase43_ledger_metadata"` row and passes only real `query_id`/`accepted_by`/`accepted_reason` rows into Phase 40 acceptance loading. The Phase 40 loader never sees a sentinel row (which lacks those required fields and would otherwise raise). This is pinned by the named tests in Task 1.

**D43-02-G — Baseline capture is semantic+FTS-only; graph engines are disabled for the rehearsal window and graph-category golden queries are excluded from the shadow comparison (addresses cycle-2 consensus HIGH "baseline graph state is not bound to the rehearsal window").**
Source-confirmed root cause: `Settings.model_copy(update={"index_dir": ...})` (D43-02-A) redirects only the SQLite/FTS5/sqlite-vec side. The FalkorDB graph backend is built by `_create_graph_store()` (`backend/src/dotmd/ingestion/pipeline.py:161`) as `FalkorDBGraphStore(url=settings.falkordb_url, graph_name="dotmd")` — bound to `settings.falkordb_url` (`backend/src/dotmd/core/config.py:208`) and a hardcoded `graph_name="dotmd"`, independent of `index_dir`. So a baseline `DotMDService` built from the cloned settings would point SQLite at the rehearsal copy while graph engines still reach the LIVE FalkorDB container (breaking snapshot isolation), or crash if the clone lacks a graph URL (breaking completeness). D43-02-A's "entire old stack" claim is therefore incomplete for graph/hybrid queries.

Resolution chosen: **option (b) — exclude graph from the baseline pool and scope Phase 43 to semantic+FTS quality evidence — implemented by building the baseline pool graph-free at construction (D43-02-I), NOT by restricting `SearchMode` on a `DotMDService`.** Option (a) (a dedicated isolated graph name/target) is rejected because `graph_name="dotmd"` is hardcoded in `_create_graph_store()`; isolating it would require editing production startup wiring, which violates the evidence-only / no-cutover scope of this phase. The cycle-2 "restrict `SearchMode`" mechanism is rejected/superseded because the FalkorDB connect + `CREATE INDEX` write happen in the `DotMDService`/`IndexingPipeline` constructor, before any `SearchMode` gate (see D43-02-I). Option (b) is achievable entirely without constructing those classes.

Concrete mechanism (see D43-02-I for full detail):
- Baseline capture does NOT construct `DotMDService`/`IndexingPipeline`/`FalkorDBGraphStore`. It builds `SemanticSearchEngine` + `FTS5SearchEngine` directly over the rehearsal `index.db` via `build_baseline_semantic_keyword_engines` and fuses them with `collect_semantic_keyword_pool` (`fuse_results({"semantic": ..., "keyword": ...})`). No graph-direct/graph-enrichment engine exists in the baseline pool, so none can fire and no FalkorDB connection is opened or `CREATE INDEX` write issued during baseline pool construction. This is an invariant by construction, not a runtime monitor (per AGENTS feedback "invariant by construction").
- The graph-free pool corresponds to the engines `_collect_candidate_pool` (`backend/src/dotmd/api/service.py:1373,1376`) fires for the SEMANTIC and KEYWORD paths separately; baseline capture reproduces exactly those two engine pools and fuses them, never the `GRAPH`/`HYBRID` graph-direct path (`service.py:1380`). There is no reliance on a combined "semantic+keyword" `SearchMode` value — the enum (`models.py:20-26`) has none.
- The golden-query corpus consumed for the shadow comparison is filtered to NON-graph queries for BOTH baseline and candidate so the two sides are compared on identical, graph-free query sets. `filter_non_graph_golden_queries(queries)` uses an EXPLICIT, DETERMINISTIC criterion against the real golden-query schema (fields `category`, `expected_engines`, `primary_surface`; verified in `backend/devtools/surreal_golden_queries.jsonl`): a query is graph-category and EXCLUDED iff its `expected_engines` list contains `"graph_direct"` OR its `primary_surface` equals `"graph_entity"`. Against the checked-in 16-query corpus this excludes exactly `sq-004`, `sq-009`, `sq-010`, `sq-015`, `sq-016` (the five rows carrying `graph_direct` and/or `graph_entity`) and retains the other 11; `category` alone is NOT the deciding field (e.g. `category: "hybrid"` rows `sq-011`/`sq-012` are retained because they carry no `graph_direct`/`graph_entity`). The excluded query ids and count are recorded in `source-capture.json` and the shadow summary so a graph-category regression is visibly out-of-scope rather than silently missed.
- The runner exposes this as a fail-closed contract: the produced `source-capture.json` carries an explicit `graph_engines_disabled: true` / `excluded_query_categories: ["graph"]` field plus the excluded query ids/count so 43-03's acceptance ledger and golden-query set reflect the exclusion.

**D43-02-H — `--candidate-config-json` has an explicit fail-closed field schema separating engine-override params from capture-loop params (addresses cycle-2 MEDIUM "`--candidate-config-json` schema underspecified", source-confirmed).**
Source-confirmed: `build_surreal_native_engine_overrides()` (`backend/src/dotmd/search/surreal_native.py:18`) accepts ONLY `embedding_dimension` (required `int`) and `hnsw_ef` (`int`, default `DEFAULT_HNSW_EF`). `top_k` and `pool_size` are NOT engine-construction params — they are inputs to the capture search loop. D43-02-D conflated the two routes. `load_candidate_config(path)` is therefore given an explicit schema:
- `embedding_dimension`: required, positive `int` → fed to `build_surreal_native_engine_overrides(..., embedding_dimension=...)`.
- `hnsw_ef`: optional, positive `int`, defaults to `DEFAULT_HNSW_EF` → fed to `build_surreal_native_engine_overrides(..., hnsw_ef=...)`.
- `top_k`: required, positive `int` → consumed by the capture search loop only (NOT passed to `build_surreal_native_engine_overrides`).
- `pool_size`: required, positive `int` → consumed by the capture search loop only (NOT passed to `build_surreal_native_engine_overrides`).
`load_candidate_config()` validates every present field is a positive integer of the correct type, applies the `hnsw_ef` default when absent, raises a `ValueError` naming the offending field on a non-positive/non-integer value or a missing required field, and REJECTS unknown keys (fail-closed) so malformed target-derived tuning cannot silently run. Pinned by named tests in Task 1.
