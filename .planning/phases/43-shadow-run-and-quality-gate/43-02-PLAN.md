<objective>
Create the Phase 43 shadow-run runner that ties Phase 40 evaluation, Phase 41 source-capture evidence, Phase 42 explicit Surreal overrides, and Phase 43 metrics into one bounded artifact bundle.

Purpose: The milestone needs old-stack baseline evidence and Surreal candidate evidence on the same production-derived window without adding cutover wiring or a fallback backend. This runner gives executors one reproducible command surface for that evidence.
Output: A tested devtool runner, focused runner tests, and an operator runbook.
</objective>

## Plan-Level Decisions

**D43-02-A — Baseline capture binds the old stack to the rehearsal path by constructing a fresh `DotMDService` from a path-overridden `Settings` clone; it never mutates live config (addresses consensus HIGH "baseline capture mechanics underspecified").**
The runner builds the baseline service exactly like `surreal_migration_runner.py` builds its config: explicit, local, single-purpose. Concrete mechanism:
- Load base settings via `load_settings()` (43-02 already imports the service layer), then produce an isolated clone with `Settings.model_copy(update={"index_dir": Path(baseline_rehearsal_path)})`. `Settings.index_db_path` derives from `index_dir` as `index_dir / "index.db"`, so overriding `index_dir` points the entire old stack (SQLite metadata + FTS5 + sqlite-vec) at the copied snapshot without touching env vars, the live TOML, or the production volume.
- Pass the cloned settings into a fresh `DotMDService(settings=cloned_settings)` constructed once per capture process (never per query, per AGENTS.md "never reload indexes per request"). Read-only search calls (`_collect_candidate_pool` with no engine overrides) produce baseline `EvalResult` rows.
- The runner does NOT call `Settings.validate_for_runtime()` on the clone (that method requires the production `/dotmd-index` and `/mnt` constants and would reject a rehearsal path); plain `DotMDService` construction does not invoke runtime validation, so the rehearsal index loads cleanly.
- No write path is invoked: no `dotmd index`, no trickle start, no `index --force`. Baseline capture is search-only against an already-built copied index.

**D43-02-B — Rehearsal-path isolation is enforced, not assumed (addresses OpenCode HIGH isolation requirement).**
Before constructing the baseline service, the runner enforces, fail-closed:
1. `--baseline-rehearsal-path` is required for `--capture-baseline`; absence is a hard error.
2. The resolved rehearsal path MUST NOT equal (or be a parent/child of) the production `DOTMD_INDEX_DIR`. The runner reads the production index dir from the environment (`DOTMD_INDEX_DIR`) and from the `RUNTIME_INDEX_DIR` constant (`/dotmd-index`) in `dotmd.core.config`, resolves both with `Path.resolve()`, and refuses to run if the rehearsal path resolves to, contains, or is contained by either. This prevents accidentally pointing baseline capture at the live volume.
3. The rehearsal `index.db` MUST pass `PRAGMA integrity_check` (opened read-only via `sqlite3.connect("file:...?mode=ro", uri=True)`) before any search call. A failing or `ok`-absent integrity result is a hard error. This proves the copied snapshot is a real, intact SQLite database rather than a truncated/symlinked stand-in.

**D43-02-C — Rehearsal-path contract: a valid copy is a real intact SQLite snapshot, not a symlink (addresses cross-cutting "Rehearsal Path Contract" gap).**
A valid rehearsal path is a directory containing an `index.db` that (a) is a regular file, not a symlink (`Path.is_symlink()` is False), (b) is not the production index dir per D43-02-B, and (c) passes `PRAGMA integrity_check`. The runbook documents that operators produce it with the SQLite backup API or a filesystem copy (`cp -r` of a stopped-container volume snapshot), consistent with the Phase 38 copied-snapshot evidence policy. The runner validates the contract; it does not produce the copy.

**D43-02-D — Flag surface is reduced by folding target-derived tuning knobs into a single `--candidate-config-json` (resolves the divergent "split vs reduce" view in favor of reduction; addresses "27-flag scope creep").**
`--embedding-dimension`, `--hnsw-ef`, `--top-k`, and `--pool-size` are removed as standalone flags. They are read from a `--candidate-config-json` descriptor (a small JSON file the operator derives from the prepared Surreal target), because they are properties of the candidate target, not runner behavior. The runner keeps the connection flags (`--target-url`, `--target-namespace`, `--target-database`) and the orchestration/artifact flags. This keeps 43-02 a single plan (no split) while shrinking the CLI surface and tying retrieval tuning to the target. `build_surreal_native_engine_overrides(connection, settings, embedding_dimension=..., hnsw_ef=...)` receives `embedding_dimension` and `hnsw_ef` from the parsed candidate-config descriptor.

**D43-02-E — `--metrics-replay-queries` is a `type=Path` JSONL descriptor (addresses "type ambiguity" concern).**
`--metrics-replay-queries` is pinned to `type=Path` pointing at a JSONL file whose rows describe the larger production-derived replay window used for latency/build/store/memory evidence (distinct from the 16-query golden quality corpus). The runner loads it with the same strict line-numbered JSON/JSONL validation used for other artifacts and fails closed on malformed rows or a missing file when metric capture is requested.

**D43-02-F — Sentinel stripping is proven by named test cases, never reaching the Phase 40 loader (addresses Codex HIGH on sentinel/Phase 40-loader interaction).**
`load_shadow_acceptance_ledger()` strips every `record_type="phase43_ledger_metadata"` row and passes only real `query_id`/`accepted_by`/`accepted_reason` rows into Phase 40 acceptance loading. The Phase 40 loader never sees a sentinel row (which lacks those required fields and would otherwise raise). This is pinned by the named tests in Task 1.
</decisions>
