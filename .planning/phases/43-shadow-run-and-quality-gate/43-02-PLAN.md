<objective>
Create the Phase 43 shadow-run runner that ties Phase 40 evaluation, Phase 41 source-capture evidence, Phase 42 explicit Surreal overrides, and Phase 43 metrics into one bounded artifact bundle.

Purpose: The milestone needs old-stack baseline evidence and Surreal candidate evidence on the same production-derived window without adding cutover wiring or a fallback backend. This runner gives executors one reproducible command surface for that evidence.
Output: A tested devtool runner, focused runner tests, and an operator runbook.

   **Stale-destination pre-copy delete (source-grounded, cycle-5 MEDIUM #3).** Source-confirmed: `Graph.copy(clone)` simply executes `COPY_CMD self.name clone` with NO pre-existence check on `clone` (`graph.py:149-161`), so an interrupted prior run that left `dotmd_shadow_baseline` behind would make a re-run non-idempotent (server-side copy error or a merge into the stale graph). `copy_baseline_graph` therefore performs a CONTROLLED pre-copy cleanup BEFORE calling `.copy(...)`: (i) first call `enforce_baseline_graph_isolation(baseline_graph)` so the production-graph guard still refuses `"dotmd"` even on the delete path (the pre-copy delete can NEVER target the production graph); (ii) call `client.list_graphs()` and, if `baseline_graph` is already present, call `client.select_graph(baseline_graph).delete()` (`GRAPH.DELETE`) to remove the stale isolate; (iii) then `client.select_graph(source_graph).copy(baseline_graph)` into the now-clean destination. This is the cleaner of the two options (pre-delete vs. early-refuse) because the isolate is a throwaway by definition and a refusal would force manual operator cleanup before every re-run; the production-name guard makes the pre-delete safe. The baseline `DotMDService` is then built (D43-02-A) over a clone whose `falkordb_graph_name` is set to the isolated baseline graph, so `FalkorDBGraphStore.__init__`'s `CREATE INDEX` writes and ALL baseline graph reads hit the isolated COPY, never production. `teardown_baseline_graph(handle)` calls `Graph.delete()` on the isolated graph after capture. Both run inside a try/finally so the copy is always torn down even on failure. Pinned by `test_copy_baseline_graph_deletes_stale_destination_before_copy` (a `list_graphs()` returning a pre-existing `dotmd_shadow_baseline` causes a `GRAPH.DELETE` on the isolate BEFORE the `.copy(...)` call, and the production `source_graph` is never deleted) and `test_copy_baseline_graph_refuses_production_destination` (`enforce_baseline_graph_isolation` still raises when `baseline_graph == "dotmd"`, so the stale-destination delete path can never hit production).

**D43-02-B — Rehearsal-path isolation is enforced, not assumed (addresses OpenCode HIGH isolation requirement).**
Before constructing the baseline service, the runner enforces, fail-closed:
1. `--baseline-rehearsal-path` is required for `--capture-baseline`; absence is a hard error.
2. The resolved rehearsal path MUST NOT equal (or be a parent/child of) the production `DOTMD_INDEX_DIR`. The runner reads the production index dir from the environment (`DOTMD_INDEX_DIR`) and from the `RUNTIME_INDEX_DIR` constant (`/dotmd-index`) in `dotmd.core.config`, resolves both with `Path.resolve()`, and refuses to run if the rehearsal path resolves to, contains, or is contained by either. 3. The rehearsal `index.db` MUST pass `PRAGMA integrity_check` (opened read-only via `sqlite3.connect("file:...?mode=ro", uri=True)`) before any search call. A failing or `ok`-absent integrity result is a hard error. This proves the copied snapshot is a real, intact SQLite database rather than a truncated/symlinked stand-in.

**D43-02-C — Rehearsal-path contract: a valid copy is a real intact SQLite snapshot, not a symlink (addresses cross-cutting "Rehearsal Path Contract" gap).**
A valid rehearsal path is a directory containing an `index.db` that (a) is a regular file, not a symlink (`Path.is_symlink()` is False), (b) is not the production index dir per D43-02-B, and (c) passes `PRAGMA integrity_check`. The runbook documents that operators produce it with the SQLite backup API or a filesystem copy (`cp -r` of a stopped-container volume snapshot), consistent with the Phase 38 copied-snapshot evidence policy. The runner validates the contract; it does not produce the copy. The runner records the rehearsal `index.db` byte size and mtime in the produced `source-capture.json` alongside the integrity result for auditability, AND (per the cycle-3 fail-closed requirement, D43-02-J) compares the rehearsal index's chunk count, embedding count, chunk strategy, and embedding model against the operator-supplied `source-capture-expected.json` and aborts before baseline capture on any mismatch — so a stale/truncated/wrong copy is not merely visible but rejected. **D43-02-E — `--metrics-replay-queries` is a `type=Path` JSONL descriptor with a defined per-row schema (addresses "type ambiguity" concern + cycle-3 LOW "`--metrics-replay-queries` per-row JSONL schema undefined").**

The runbook documents this minimal row schema (D43-02-E).

**D43-02-F — Sentinel stripping is proven by named test cases, never reaching the Phase 40 loader, AND the stripped rows are wired into `run_eval` via a temp acceptance file (addresses Codex HIGH on sentinel/Phase 40-loader interaction; pins the cycle-5 MEDIUM "sentinel→`run_eval` acceptance wiring not pinned").**

The Phase 40 loader never sees a sentinel row (which lacks those required fields and would otherwise raise).

**Concrete wiring (source-grounded, cycle-5 MEDIUM #1).** Source-confirmed: `run_eval(EvalRunnerConfig)` calls `_load_acceptances(config.acceptance)` DIRECTLY on the file path (`backend/devtools/surreal_eval_runner.py:159`; `EvalRunnerConfig.acceptance: Path | None` at `:35`), and `_load_acceptances` (`:48-82`) requires `accepted_by`/`accepted_reason` on every row — so handing it the raw `accepted-diffs.jsonl` (which contains the `phase43_ledger_metadata` sentinel row) would raise `ValueError` on the sentinel line. Therefore the runner MUST NOT pass the raw `--accepted-diffs` path into `EvalRunnerConfig.acceptance`. Instead, `run_shadow_run` calls `load_shadow_acceptance_ledger(accepted_diffs_path)` to obtain the sentinel-stripped real acceptance rows, WRITES those stripped rows to a temp acceptance file (`tempfile.NamedTemporaryFile(suffix=".jsonl", delete=False)` or a path under a `tempfile.TemporaryDirectory()`), and sets `EvalRunnerConfig.acceptance` to THAT stripped temp path. The raw `accepted-diffs.jsonl` path (sentinel-containing) is NEVER assigned to `EvalRunnerConfig.acceptance` and is NEVER handed to `_load_acceptances`. When the ledger has only the sentinel (no real acceptances), the stripped temp file is empty/zero-row and `EvalRunnerConfig.acceptance` points at that empty file (or `None`), so `_load_acceptances` receives no sentinel row. Pinned by the named tests in Task 1 (including `test_run_eval_receives_stripped_acceptance_path_not_raw_sentinel_file`).
