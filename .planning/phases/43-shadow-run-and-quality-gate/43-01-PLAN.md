## Plan-Level Decisions

**D43-01-A — Memory guardrails compare candidate-vs-baseline from a paired payload (addresses review concern: "Memory guardrail comparison baseline undefined").**
The ratio guardrails (`candidate_rss_growth_ratio`, `candidate_python_heap_growth_ratio`) are evaluated only against a paired memory payload that carries BOTH a baseline `ShadowMemoryMetrics` and a candidate `ShadowMemoryMetrics`. A single unpaired memory payload cannot satisfy `validate_shadow_metric_bundle()` for ratio evaluation. `evaluate_shadow_memory_guardrails(baseline, candidate, guardrails)` computes `candidate.max_rss_bytes / baseline.max_rss_bytes` and `candidate.peak_python_heap_bytes / baseline.peak_python_heap_bytes`, then passes only when each ratio is at or below the configured ratio OR the absolute candidate-minus-baseline delta is at or below the configured slack bytes (slack absorbs small-corpus noise where ratios over-trigger). Both `baseline` and `candidate` memory payloads are required fields of `ShadowMetricBundle`; absence of either is a fail-closed `ValueError`.

**D43-01-B — Guardrail constant rationale is documented in the dataclass (addresses review concern: "Arbitrary guardrail constants").**
`ShadowMemoryGuardrails` carries a class docstring documenting why each default exists: the `1.25` ratios mean "candidate may use up to 25% more peak RSS / Python heap than the old stack for the same replay window before the guardrail trips" (a deliberately loose first-cutover tolerance, not a tuned SLO); the `268435456`-byte RSS slack (256 MiB) and `134217728`-byte Python-heap slack (128 MiB) exist so that on a small bounded replay window an absolute delta below process-level allocator/runtime noise does not trip the ratio even when the ratio math exceeds `1.25`. These are evidence-gate tolerances for one bounded shadow window, explicitly re-tunable in a later cutover phase; they are not production memory budgets.

**D43-01-C — `capture_shadow_memory_metrics` is an optional measurement helper, not the contract's measurement owner (resolves divergent reviewer view on helper placement).**
The bundle/guardrail/validation/JSON surface is the contract. `capture_shadow_memory_metrics(label, callable_)` is provided as a convenience so the 43-02 runner can produce a `ShadowMemoryMetrics` payload, but the runner is the authoritative caller of measurement. The helper must call `tracemalloc.start()` before invoking the callable and `tracemalloc.stop()` after, so `peak_python_heap_bytes` is a real traced value and never a silent zero that would pass presence checks without evidence.
</decisions>
