## Plan-Level Decisions

**D43-03-A — An automated pre-flight validates the Phase 41 Surreal candidate target BEFORE any capture (addresses OpenCode HIGH "Phase 41 Surreal target prerequisite is unvalidated").**
Task 1 runs an automated pre-flight, before baseline or candidate capture, that confirms the transform-first Surreal candidate target prepared from Phase 41 evidence (a) exists at the configured `--target-url` and (b) is queryable. The pre-flight opens the Surreal connection and runs a basic retrieval smoke (a bounded `SELECT`/count over the imported chunk table for the active chunk strategy) using the same explicit Surreal seam the candidate capture uses. If the target is missing, unreachable, or returns zero imported records, the pre-flight exits nonzero and Task 1 stops and records the exact missing input (no recompute, no live mutation). This converts the prior manual "stop and record missing input" clause into an automated gate.

**D43-03-D — The pre-flight binds the candidate target to the operator-supplied immutable expected manifest, not just reachability (addresses cycle-2 MEDIUM "candidate preflight identity binding"; manifest input/output split per cycle-3 D43-02-K).**
Reachability + nonzero records alone (D43-03-A) cannot tell a correct target from a stale namespace/database that happens to contain unrelated records. The Task 1 pre-flight therefore passes `--source-capture-manifest-json` (pointing at the operator-supplied immutable `source-capture-expected.json`, NOT the runner-produced `source-capture.json`) to `preflight_candidate_target(connection, settings, candidate_config, expected_manifest)` (extended in Plan 43-02 D43-03-D), which asserts the target identity matches the expected manifest BEFORE capture: chunk strategy, embedding model, import id, and expected imported record counts. Because the validated input is never rewritten by the runner (D43-02-K), validation cannot pass against an artifact the runner later mutates. A reachable, non-empty target whose identity does not match the expected manifest fails the pre-flight and stops the run with the exact mismatch recorded, so a stale or wrong target cannot silently produce candidate evidence.

**D43-03-E — Phase 43 shadow evidence is symmetric and full-fidelity: the graph is isolated on BOTH sides and the FULL 16-query corpus runs on both (carries Plan 43-02 D43-02-L into the evidence bundle; SUPERSEDES the cycle-3 graph-exclusion framing).**
Under Option A (D43-02-L), the baseline binds to an isolated FalkorDB graph COPY (GRAPH.COPY of the production `dotmd` graph into `dotmd_shadow_baseline`, bound via the new `falkordb_graph_name` config knob) and the candidate uses its isolated Phase 41 Surreal target. Both sides therefore query isolated graph copies for the rehearsal window — never the production `dotmd` graph, and never "no graph at all." Consequently this plan's golden-query set is the COMPLETE 16-query corpus for BOTH baseline and candidate, including the `graph-entity` category (`sq-004/009/010`) AND the bilingual `mixed-ru-en` rows (`sq-015/016`, the project's core multilingual use case); `run_eval()` runs with `require_complete_category_coverage=True` and no category is dropped. The produced `source-capture.json` records the isolated baseline graph name used for the GRAPH.COPY (proving the production graph was not queried), and `run_shadow_run` asserts the production `index.db` and production `dotmd` graph are untouched. There is NO non-graph subset, NO `graph_engines_disabled`/`excluded_query_categories` marker, and NO `filter_non_graph_golden_queries`; a graph-category regression now surfaces as a real classified diff in `shadow-diffs.jsonl` rather than being out-of-scope.

**D43-03-B — Task 1 verification validates artifact VALIDITY, not just non-emptiness (addresses "test -s only checks non-empty, not validity" and "rehearsal-path-is-a-real-copy check").**
Beyond `test -s`, Task 1 verification (1) loads `baseline-results.jsonl` and `candidate-results.jsonl` through `load_eval_results()` so corrupt/malformed JSONL fails the gate, (2) asserts baseline and candidate `query_id` sets match over the FULL 16-query corpus (all categories incl. graph-entity and mixed-ru-en — no subset, D43-03-E), (3) parses the produced `source-capture.json` as JSON and asserts it records an isolated baseline graph name that is not `"dotmd"`, (4) runs the runner's rehearsal-path-isolation check (D43-02-B/C: real intact non-symlinked `index.db` that passes `PRAGMA integrity_check` and does not overlap the production index dir) plus the baseline-graph-isolation check (D43-02-L: baseline graph name `!= "dotmd"`), and (5) relies on the runner's fail-closed rehearsal-identity comparison (D43-02-J: rehearsal `index.db` chunk count, embedding count, chunk strategy, and embedding model must match `source-capture-expected.json`) so a symlink-to-production, truncated, or stale/wrong-identity copy cannot pass as a rehearsal source.

**D43-03-C — Critical acceptance rules are gate-level, not buried in task prose; `--verify-only` actively detects tampering (addresses LOW "critical rules buried in Task 3 action" + cycle-3 LOW "`--verify-only` does not explicitly regenerate-and-compare `shadow-diffs.jsonl`").**
"No regression accepted without an explicit reason" and "raw `shadow-diffs.jsonl` classifications are never hand-edited" are restated in this plan's `<verification>` and `<success_criteria>`, not only inside the Task 3 `<action>`, so the executor and reviewer treat them as gate conditions. Tamper-detection is made active, not implicit: `--verify-only` (Plan 43-02) REGENERATES `shadow-diffs.jsonl` from the on-disk baseline/candidate results via `run_eval()` and COMPARES it against the on-disk `shadow-diffs.jsonl`; any divergence (a hand-edited raw classification row) is a hard failure (nonzero exit). The Task 2/Task 3 `--verify-only` invocations therefore prove the raw classifications were not altered, not merely that the file is non-empty.
