## Plan-Level Decisions

**D43-03-A — An automated pre-flight validates the Phase 41 Surreal candidate target BEFORE any capture (addresses OpenCode HIGH "Phase 41 Surreal target prerequisite is unvalidated").**
Task 1 runs an automated pre-flight, before baseline or candidate capture, that confirms the transform-first Surreal candidate target prepared from Phase 41 evidence (a) exists at the configured `--target-url` and (b) is queryable. The pre-flight opens the Surreal connection and runs a basic retrieval smoke (a bounded `SELECT`/count over the imported chunk table for the active chunk strategy) using the same explicit Surreal seam the candidate capture uses. If the target is missing, unreachable, or returns zero imported records, the pre-flight exits nonzero and Task 1 stops and records the exact missing input (no recompute, no live mutation). This converts the prior manual "stop and record missing input" clause into an automated gate.

**D43-03-D — The pre-flight binds the candidate target to the source-capture manifest identity, not just reachability (addresses cycle-2 MEDIUM "candidate preflight identity binding").**
Reachability + nonzero records alone (D43-03-A) cannot tell a correct target from a stale namespace/database that happens to contain unrelated records. The Task 1 pre-flight therefore passes `--source-capture-manifest-json` to `preflight_candidate_target(connection, settings, candidate_config, expected_manifest)` (extended in Plan 43-02 D43-03-D), which asserts the target identity matches the manifest BEFORE capture: chunk strategy, embedding model, import id, and expected imported record counts. A reachable, non-empty target whose identity does not match the manifest fails the pre-flight and stops the run with the exact mismatch recorded, so a stale or wrong target cannot silently produce candidate evidence.

**D43-03-E — Phase 43 shadow evidence is semantic+FTS-only; graph-category golden queries are excluded and the exclusion is recorded in the acceptance ledger (carries Plan 43-02 D43-02-G into the evidence bundle).**
Because the rehearsal `index.db` snapshots only SQLite/FTS5/sqlite-vec and the FalkorDB graph backend is bound by `falkordb_url` (not `index_dir`), baseline capture disables graph engines (D43-02-G). This plan's golden-query set used for the shadow comparison is therefore the NON-graph subset for both baseline and candidate, and the excluded graph-category query ids/count plus a `graph_engines_disabled` / `excluded_query_categories` marker are recorded in `source-capture.json`, `accepted-diffs.jsonl` (sentinel), and `shadow-summary.md`. The acceptance ledger explicitly states that graph-category queries are out-of-scope for Phase 43 evidence, so no graph-category regression is silently treated as a pass.

**D43-03-B — Task 1 verification validates artifact VALIDITY, not just non-emptiness (addresses "test -s only checks non-empty, not validity" and "rehearsal-path-is-a-real-copy check").**
Beyond `test -s`, Task 1 verification (1) loads `baseline-results.jsonl` and `candidate-results.jsonl` through `load_eval_results()` so corrupt/malformed JSONL fails the gate, (2) asserts baseline and candidate `query_id` sets match, (3) parses `source-capture.json` as JSON, and (4) runs the runner's rehearsal-path-isolation check (D43-02-B/C: real intact non-symlinked `index.db` that passes `PRAGMA integrity_check` and does not overlap the production index dir) so a symlink-to-production or truncated copy cannot pass as a rehearsal source.

**D43-03-C — Critical acceptance rules are gate-level, not buried in task prose (addresses LOW "critical rules buried in Task 3 action").**
"No regression accepted without an explicit reason" and "raw `shadow-diffs.jsonl` classifications are never hand-edited" are restated in this plan's `<verification>` and `<success_criteria>`, not only inside the Task 3 `<action>`, so the executor and reviewer treat them as gate conditions.
