# Phase 38-01 Plan: Standalone SurrealDB Migration Proof

## Goal

Prove that dotMD can use a standalone SurrealDB 3.1.x server as the migration
target without carrying forward embedded SurrealKV-specific workarounds.

## Decisions

- Start from `main` in a clean branch: `milestone/v1.8-standalone-surrealdb`.
- Use shared host service `/opt/docker/surrealdb` with data in `/srv/surrealdb`.
- Use WebSocket RPC for Python client work: `ws://127.0.0.1:8000/rpc` on host,
  `ws://surrealdb:8000/rpc` from containers.
- Start with monolithic `embeddings`; sharding is a fallback only.
- Any operation silent for more than 120 seconds must be instrumented before it
  is allowed into the migration path.

## Tasks

- [x] Create clean worktree from `main`.
- [x] Deploy standalone SurrealDB 3.1.4 with RocksDB.
- [x] Add standalone smoke CLI with read/write probe.
- [x] Add minimal standalone SurrealDB connection/schema foundation.
- [x] Port only the migration transforms required for SQLite/Falkor source data.
- [x] Run microbench before any full quality gate.

## First Gate

Run:

```bash
cd backend
set -a
. /opt/docker/surrealdb/.env
set +a
uv run python devtools/surreal_standalone_smoke.py --write-probe
```

Expected: authenticated version report and successful temporary record
create/read/delete in less than a few seconds.
