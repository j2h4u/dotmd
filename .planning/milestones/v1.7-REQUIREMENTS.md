# Requirements Archive: v1.7 Storage Simplification

**Archived:** 2026-06-12
**Status:** Shipped

For current requirements, see `.planning/REQUIREMENTS.md`.

## Goal

Evaluate whether embedded SurrealDB can replace the current
SQLite/sqlite-vec/FTS5 + FalkorDB split with one embedded storage layer while
migrating existing production data wherever safe instead of recomputing chunks,
embeddings, or extracted entities on CPU.

## Requirements

### Storage

- [x] **STOR-01**: dotMD can model its current persistent data in embedded
  SurrealDB: documents, source units, chunks, embeddings, entities, relations,
  feedback, cursors, and checkpoints.

- [x] **STOR-02**: The SurrealDB prototype can execute the retrieval paths dotMD
  depends on: full-text, vector, graph-direct entity retrieval, and hybrid/RRF
  fusion.

- [x] **STOR-03**: The spike measures how much current production data can be
  migrated from SQLite/sqlite-vec/FalkorDB without CPU-heavy rechunking,
  reembedding, or re-extraction.

- [x] **STOR-04**: The spike produces a recommendation to migrate, defer, or
  reject SurrealDB, including operational notes for backup/restore,
  locking/concurrency, and rollback.

## Outcomes

| Requirement | Phase | Status | Outcome |
|-------------|-------|--------|---------|
| STOR-01 | Phase 38 | Complete | Thin SurrealDB schema and import proof model current persistent data. |
| STOR-02 | Phase 38 | Complete | Retrieval paths were exercised; migrate gate failed on weighted FTS and hybrid/RRF evidence. |
| STOR-03 | Phase 38 | Complete | Transform-first migration proof preserved stored rows without default recomputation. |
| STOR-04 | Phase 38 | Complete | Final recommendation is reject for the first compatibility/parity prototype. |

## Decision

The milestone is complete because it answered the decision question. The next
milestone should not try to productize exact compatibility with the old stack;
it should build SurrealDB-native retrieval, prove quality through evaluation,
cut over, and remove the legacy stack.
