# Milestone v1.5: Telegram Source Adapter

**Status:** SHIPPED 2026-05-08
**Phases:** 27-31
**Total Plans:** 13 completed, 1 deferred phase

## Overview

v1.5 made Telegram a first-class application source boundary for dotMD. The
milestone introduced retained resource bindings, a generic application source
provider contract, Telegram ingestion through the existing `mcp-telegram`
runtime, and a live search/read/drill smoke against real Telegram messages.

Phase 30 incremental Telegram sync was intentionally deferred to Backlog 999.30
so it can be implemented through the next unified source architecture line
instead of as a Telegram-only legacy branch.

## Phases

### Phase 27: Resource bindings and retained artifacts foundation

**Goal:** Separate active public source visibility from retained content and
derived artifacts.
**Plans:** 4/4 complete

Delivered:
- SQLite resource binding state with active/inactive visibility.
- Source-document backfill into active bindings.
- Public search/read/drill active-binding gates.
- Retained artifact lifecycle docs and regression proof.

### Phase 28: Application source provider contract

**Goal:** Define the generic provider boundary for application-backed sources.
**Plans:** 4/4 complete

Delivered:
- Application source payload models and provider protocol.
- Source checkpoint and source-unit fingerprint persistence.
- Deterministic fixture provider for cursor/window behavior.
- Telegram contract notes for the `mcp-telegram` integration boundary.

### Phase 29: Telegram adapter MVP ingestion

**Goal:** Ingest Telegram source units into dotMD through `mcp-telegram`.
**Plans:** 4/4 complete

Delivered:
- `mcp-telegram` source export contract.
- dotMD Telegram provider backed by the daemon socket.
- Telegram source-unit ingestion with FTS5/sqlite-vec writes in the same transaction.
- Telegram read/drill support and fixture regression coverage.

### Phase 30: Incremental Telegram sync and reuse

**Status:** Deferred to Backlog 999.30.

Reason:
- The behavior belongs in the unified source lifecycle and federated-search
  architecture line, not as a Telegram-specific legacy path.

Carried forward:
- Repeated sync should process only new or changed source units.
- Unchanged history should not be rechunked/reembedded.
- Sync reporting should expose discovered, new, changed, rebound, skipped,
  hidden, failed, and reused counts where practical.

### Phase 31: Telegram search/read/drill smoke

**Goal:** Verify the public workflow:
`search(query) -> ref -> drill(ref) / read(ref, start, end)`.
**Plans:** 1/1 complete

Delivered:
- Existing MCP e2e smoke passed: `36 passed`.
- Live Telegram ingest processed 100 messages with `failed_units=0`.
- Live search found real Telegram text and returned
  `telegram:dialog:-1003897013523:message:80`.
- `drill(ref)` and `read(ref)` round-tripped through dotMD alone.
- Fixed Telegram search result hydration so message-level refs are returned
  when chunk provenance identifies a single Telegram source unit.

## Milestone Summary

**Key decisions:**
- Telegram remains an application source backed by `mcp-telegram`; dotMD does
  not own Telegram auth or instantiate a Telegram API client.
- Public source identity remains ref-first.
- Telegram public message refs use
  `telegram:dialog:<dialog_id>:message:<message_id>`.
- Phase 30 was deferred to Backlog 999.30 to avoid creating a second legacy
  source plane before the unified source architecture work.

**Issues resolved:**
- Telegram data can be represented as source documents/source units without
  pretending it is a filesystem file.
- Public search/read/drill now works for Telegram message refs.
- Retained inactive artifacts stay internal and are hidden from public output.

**Issues deferred:**
- Full incremental Telegram sync/reuse.
- Telegram federated search through the native MCP Telegram FTS path.
- Edit/delete/TTL lifecycle policy.
- Attachments/media indexing.

**Technical debt:**
- Source registry, lifecycle/config/auth/cursor, federated search candidates,
  and filesystem/Telegram unification are captured as Backlog 999.26-999.31 for
  the next milestone.
