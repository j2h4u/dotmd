<objective>
Verify and lock the TG-04 ref consistency invariant, and add the missing
`rebound_units` counter for TG-03.

TG-04 situation (verified by code inspection):
- Local search already returns message-level refs via `ChunkProvenance.ref =
  f"{namespace}:{unit.unit_ref}"` = `"telegram:dialog:<id>:message:<id>"`
  (pipeline.py line 845).
- Federated `search_native` also returns `"telegram:dialog:<id>:message:<id>"`
  (telegram_provider.py line 246).
- `ResourceBinding.ref` is dialog-level ("telegram:dialog:<id>") by model
  validator design — this is an internal binding lookup key, not the public ref.
- TG-04 is satisfied by construction. This plan adds a regression test to pin
  the invariant so it cannot silently regress.

TG-03 situation:
- `ApplicationSourceIngestResult` has 8 fields but is missing `rebound_units`.
- "Rebound" = a unit that was previously skipped (unchanged fingerprint) but
  whose binding was re-activated after having been inactive.
- The pipeline's second loop (batch.changes) already calls upsert_resource_binding
  for all units including skipped ones — so binding refresh for D-05 is already
  correct. rebound_units is a reporting counter only.

Purpose: TDD locks these two correctness properties as failing tests first,
then adds minimal implementation to make them green.
Output: `rebound_units` counter wired; TG-04 invariant pinned by regression test.
</objective>

    TG-03 behavior:
    - `ApplicationSourceIngestResult` has a `rebound_units: int = 0` field
    - Default value is 0
    - CLI `telegram ingest` output includes `rebound_units=N` and `reused_units=N`

    TG-04 regression test behavior:
    - `public_ref_for_unit(unit)` returns `"telegram:dialog:<id>:message:<id>"`
    - `ChunkProvenance.ref` for a Telegram chunk equals `"telegram:dialog:<id>:message:<id>"`
    - `search_native` SearchCandidate ref equals `"telegram:dialog:<id>:message:<id>"`
    - All three are equal for the same message

    rebound_units detection (pipeline):
    - Before calling upsert_resource_binding, check if an existing binding has active=False
    - If existing binding.active is False → increment result.rebound_units
    - Uses get_resource_binding(namespace, resource_ref=change.document.document_ref)
      because upsert_resource_binding uses resource_ref=document.document_ref (dialog-level)
    - NOTE: rebound detection applies to all changes in batch.changes; new units without
      an existing binding return None from get_resource_binding and are not incremented
  </behavior>

      In `backend/src/dotmd/ingestion/pipeline.py`, locate `ApplicationSourceIngestResult`
      (line ~102). Add after `reused_units: int = 0`:
        `rebound_units: int = 0`

    Step 2 — Add rebound detection in `_ingest_application_source`:
      In the second loop (line ~583, `for change in batch.changes:`), before
      `upsert_resource_binding`, check:
        ```python
        existing_binding = self._metadata_store.get_resource_binding(
            change.unit.namespace,
            change.unit.unit_ref,  # unit-level resource_ref for Telegram
        )
        if existing_binding is not None and not existing_binding.active:
            result.rebound_units += 1
        ```
      Note: `get_resource_binding` is in `backend/src/dotmd/storage/metadata.py`.
      Read its signature first to confirm parameter names.

      after `ingest_application_source_runtime` (line ~480):
      Add `rebound_units={result.rebound_units}` and `reused_units={result.reused_units}`
      to the output string. Place `rebound_units` after `changed_units` and
      `reused_units` at the end (after `failed_units`).
  </implementation>
