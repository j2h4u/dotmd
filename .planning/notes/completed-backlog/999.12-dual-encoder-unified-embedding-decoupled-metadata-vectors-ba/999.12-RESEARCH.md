# Phase 999.12 Research: Dual-encoder Unified Embedding — Decoupled Metadata Vectors

**Researched:** 2026-04-27
**Phase:** 999.12 — Dual-encoder unified embedding / decoupled metadata vectors

---

## Research Question

What do I need to know to PLAN Phase 999.12 well?

The phase decouples metadata (title+tags) from chunk body embeddings so that
metadata-only changes require 1 TEI call per file instead of N TEI calls per
chunk. It introduces a `vec_components` table, replaces the current enriched
`e_fused` vector in vec0, adds a `search_log` table, and introduces a
`meta_tracker` fingerprint.

---

## 1. Current State: What Gets Touched

### enrich_with_title_and_tags (content_handlers.py:51)

The current `enrich_with_title_and_tags()` function prepends `title\ntags\n\n`
to every chunk's text before encoding. This is called via the handler registry
in `_enrich_for_embedding()` (pipeline.py:441). The result is hashed as
`text_hash` so any title/tag change → different hash for all chunks → 0 cache
hits → full re-embed.

**What changes:** This function is removed from the embedding path entirely.
The enrichment registry entry still exists for FTS5 (FTS5 enrichment is a
separate path that does NOT use text_hash). The embedding path in
`_embed_chunks()` now calls `chunk.text` directly (no enrichment).

### _embed_chunks() (pipeline.py:443-519)

Currently: enriches text → computes text_hash on enriched text → cache lookup
→ encode misses → store.

**What changes:**
- `enriched_texts` → replaced by raw `chunk.text` (no handler enrichment for embedding)
- `text_hash` computed on raw `chunk.text` only → same chunk body in different
  files now truly shares a hash → cross-strategy reuse works correctly
- Adds a new code path: for each batch, also stores `e_text` BLOB in
  `vec_components` table
- Still writes to global `embedding_cache` (unchanged)
- Does NOT compute `e_fused` directly — `e_fused` is computed separately after
  `e_meta` is available (see fusion below)

### embed_checksum / meta_tracker (reader.py:66-88, file_tracker.py)

Currently: `embed_tracker` uses `embed_checksum` = `hash(body + kind + title + tags)`.

**What changes:**
- `embed_tracker` is removed (no longer needed: body changes trigger
  chunk_tracker; metadata changes trigger meta_tracker)
- New `meta_tracker` = `FileTracker(conn, table_name=meta_fp_table,
  checksum_fn=meta_checksum)` where `meta_checksum` = `hash(title + tags)`
  (new function in reader.py)
- `chunk_tracker` unchanged: `hash(body + kind)` → re-chunking
- Pipeline `_save_embed_fingerprint` renamed to `_save_meta_fingerprint`, uses
  `meta_tracker`

### Metadata-only change path (pipeline.py:1100-1148)

Currently: on `meta_tracker` diff → calls full `_embed_chunks(chunks)` which
re-encodes all chunk bodies via TEI.

**What changes:** The "metadata-only" fast path becomes:
1. Read `e_text` BLOBs from `vec_components` for all chunk_ids of the file
2. Compute `e_meta = encode_batch([title + " " + tags_str])` — 1 TEI call
3. For each chunk: `e_fused = normalize(w_text * normalize(e_text) + w_meta * normalize(e_meta))`
4. DELETE existing vec0 rows for those chunk_ids + INSERT new e_fused
5. Save meta fingerprint

This is O(1) TEI calls per file regardless of chunk count.

## 2. Schema: New Tables and Changes

### vec_components_<strategy>_<model>

```sql
CREATE TABLE IF NOT EXISTS vec_components_<strategy>_<model> (
    entity_id  TEXT NOT NULL,
    component  TEXT NOT NULL,  -- 'text' or 'meta'
    embedding  BLOB NOT NULL,
    PRIMARY KEY (entity_id, component)
)
```

- `text` component: keyed by `chunk_id`, BLOB of float32 vector for `embed(chunk.text)`
- `meta` component: keyed by `file_path`, BLOB of float32 vector for `embed(title + tags)`
- Two-dimensional naming follows existing convention: suffix `_<strategy>_<model>`

### vec0 (vec_chunks_<strategy>_<model>)

Stores `e_fused` instead of `embed(title + tags + chunk_text)`. Interface
unchanged: same `rowid` join with `vec_meta_<strategy>_<model>`. ANN search
over e_fused is unchanged.

### search_log

```sql
CREATE TABLE IF NOT EXISTS search_log (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    query        TEXT NOT NULL,
    timestamp    TEXT NOT NULL,
    weights_used TEXT NOT NULL,   -- JSON: {"text": 0.7, "meta": 0.3}
    top_results  TEXT NOT NULL    -- JSON array of {chunk_id, score, engine}
)
```

Log every search. Retention: cap at 10,000 rows (DELETE oldest when inserting
beyond cap) — simple, no timer needed, predictable storage.

### Schema version sentinel

Use `vec_config_<strategy>_<model>` (already exists per strategy+model):

```sql
INSERT OR REPLACE INTO vec_config_... (key, value) VALUES ('schema_version', '2')
```

On `_ensure_tables()` startup: read schema_version. If absent or != '2':
- Clear `embedding_cache` (text_hash semantics changed)
- Clear `vec_components_*` if exists
- Clear vec0, vec_meta tables
- Clear both tracker tables (chunk + meta)
- Set schema_version = '2'
- trickle rebuilds from scratch

This wipe-on-deploy migration requires no migration script — handled entirely
within `_ensure_tables()` at container startup.

## 3. Fusion Formula Implementation

```python
import struct, math

def _normalize(v: list[float]) -> list[float]:
    mag = math.sqrt(sum(x * x for x in v))
    return [x / mag for x in v] if mag > 0 else v

def _fuse(e_text: list[float], e_meta: list[float], w_text: float, w_meta: float) -> list[float]:
    nt = _normalize(e_text)
    nm = _normalize(e_meta)
    raw = [w_text * a + w_meta * b for a, b in zip(nt, nm)]
    return _normalize(raw)
```

This is pure Python using stdlib `math` — no numpy/scipy required (correct for
this CPU-only, AVX-limited environment).

### DOTMD_EMBEDDING_WEIGHTS parsing

`"text=0.7,meta=0.3"` → `{"text": 0.7, "meta": 0.3}`

Validate at startup:
- All component names present (text + meta)
- Values are floats, sum ≈ 1.0 (tolerance ±0.001)
- Error on startup, not at search time

Recomputing e_fused when weights change is cheap: read all e_text + e_meta
BLOBs from `vec_components`, fuse locally, bulk-update vec0. No TEI calls.
This can be triggered by `dotmd reweight` CLI command or detected at startup
if stored weights differ from env var.

## 4. Pipeline Initialization Changes

### New `meta_checksum` function (reader.py)

```python
def meta_checksum(path: Path) -> str:
    """Checksum for metadata-only change detection: hash(title + tags)."""
    content = read_file(path)
    frontmatter, _ = parse_frontmatter(content)
    title = str(frontmatter.get("title", ""))
    tags = frontmatter.get("tags", [])
    tags_str = ",".join(sorted(str(t) for t in tags)) if tags else ""
    return blake3.blake3(f"{title}\n{tags_str}".encode()).hexdigest()
```

### Tracker setup in `IndexingPipeline.__init__`

```python
# BEFORE (v current):
self._chunk_tracker = FileTracker(conn, table_name=chunk_fp_table, checksum_fn=chunk_checksum)
self._embed_tracker = FileTracker(conn, table_name=embed_fp_table, checksum_fn=embed_checksum)

# AFTER (v999.12):
self._chunk_tracker = FileTracker(conn, table_name=chunk_fp_table, checksum_fn=chunk_checksum)
self._meta_tracker  = FileTracker(conn, table_name=meta_fp_table,  checksum_fn=meta_checksum)
# embed_tracker removed
```

`embed_checksum` in reader.py is removed (or kept as unused dead code — remove
for clean break).

## 6. Search Path: Query Embedding

The query is encoded as `embed(query_text)` — NOT fused. This is correct per
the paper: the query vector lives in the same space as e_text (body embeddings).
e_fused shifts document vectors toward their metadata, but query vectors stay
in the text space. ANN search over e_fused against a text-space query is the
intended matching pattern.

No changes to `SemanticSearchEngine.encode()` or `search()`.

### search_log write path

In `DotMDService.search()` (api/service.py), after RRF fusion and reranking,
write one row to `search_log`:

```python
self._pipeline.log_search(
    query=query,
    weights_used=self._pipeline.get_weights(),
    top_results=[{"chunk_id": r.chunk_id, "score": r.score, "engine": r.engine}
                 for r in results[:top_k]],
)
```

`log_search` writes to `search_log` via the shared `_conn`. Non-blocking —
uses `INSERT OR IGNORE` on failure. Retention: trim when row count exceeds
10,000 (DELETE WHERE id < (SELECT MIN(id) + N FROM search_log)).

## 8. Deployment Considerations

### Rebuild time

As documented in CONTEXT.md D-14: **several days** on current hardware
(Xeon E3-1245 V2, TEI CPU-only). This is a known trade-off. The deploy plan:
- Deploy on a weekend or when degraded search is acceptable
- trickle rebuilds from scratch automatically after wipe-on-deploy
- No manual intervention needed — just `docker compose up -d` after code deploy

### Weight storage in vec_config

Store current weights in `vec_config_<strategy>_<model>`:
```sql
INSERT OR REPLACE INTO vec_config_... VALUES ('weights', '{"text":0.7,"meta":0.3}')
```

On startup, if stored weights differ from `DOTMD_EMBEDDING_WEIGHTS` env var →
trigger `_recompute_fused()` (reads all components, rewrites vec0). This is
fully local (no TEI) and fast (~seconds for a few thousand chunks).

### search_log retention

Cap at 10,000 rows. Simple and predictable. Each row is ~200-300 bytes
(JSON of top-20 results). Max table size ≈ 3 MB. Fine for sqlite.
